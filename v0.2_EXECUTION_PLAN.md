# OpenList Android Client v0.2 Execution Plan

> 本文件是 v0.2 的**可执行开发计划**，非泛泛建议。除本文件外，本轮未创建、修改、删除任何业务文件，未初始化/修改 Gradle，未写业务代码，未装依赖，未构建，未改动任何既有文档。
>
> **三项已确认关键决策（用户已明确选择）：**
> - **决策 A = B（多模块）**：v0.2 开工前先把当前单模块 `:app` 拆分为多 Gradle 模块，再开发功能。
> - **决策 B = B（上传执行）**：上传任务由 **WorkManager**（+ 前台服务通知 + Hilt Worker）承载。
> - **决策 C = A（批量策略）**：批量删除/移动/复制在客户端**逐项串行调用**后端并聚合为 `BatchOperationResult`，以满足"部分成功/部分失败"验收；与 Web 单请求行为的差异记入 Parity。

## 8. Architecture Delta from v0.1

v0.2 不改变 v0.1 分层与强制规则（UI 禁止直连 Retrofit/DAO；结果统一 `ApiResult`/`DomainError`；数据带 `instanceId`；路径走 `OpenListPathCodec`；Token 仅经 Provider/Keystore）。增量：

1. **工程物理结构**：单模块 → 多 Gradle 模块（§9，决策 A）。
2. **写链路**：新增 `FileOperationRepository`（写操作 + 逐项批量聚合 + 写后失效）。
3. **上传链路**：新增 `UploadRepository` + WorkManager `UploadWorker` + 前台服务通知（决策 B）。
4. **目录选择**：新增 `DirectoryPickerRepository`（只列目录，复用 `fsList`）。
5. **权限模型**：`SessionEntity.permission` 位掩码 + `FsListResp.write` 目录级门控。
6. **缓存一致性**：从"读缓存"扩展到"写后失效"（§18）。
7. **数据模型**：新增 `UploadTaskEntity`(+DAO)、可选 `OperationLogEntity`；域模型 `BatchOperationResult/Failure`；Room v4→v6 正式 Migration。
8. **API 表面**：`OpenListApi` 增 mkdir/rename/remove/move/copy + put/form（上传走独立 OkHttp 流式，不走标准 Retrofit body）。

## 10. API Integration Plan

### 10.1 组织方式
- **读/写 JSON 接口**：扩展现有 `OpenListApi`（Retrofit + `ApiResponse<T>`），新增 mkdir/rename/remove/move/copy。
- **上传**：**不走标准 Retrofit body**。新增 `UploadApi`（或 OkHttp 直用）以支持流式 `RequestBody` + 进度回调；`PUT api/fs/put` 走原始流，`PUT api/fs/form` 走 multipart。上传用**独立 OkHttpClient**（长超时、无 30s 读超时限制、可取消）。
- **Repository 层**：`FileOperationRepositoryImpl`、`UploadRepositoryImpl`、`DirectoryPickerRepositoryImpl` 置于 `:data:repository`。
- **Token 注入**：沿用 v0.1 `AuthInterceptor`（按 `instanceId`）。上传的独立 client **也必须挂 `AuthInterceptor`**。
- **多实例 Base URL**：沿用 `OpenListClientFactory` 按 `instanceId` 取 client；上传 client 同样按实例取。
- **路径编码**：全部复用 `OpenListPathCodec`；上传 `File-Path` header 需 **URL 编码全路径**（后端 `url.PathUnescape`）。
- **写错误统一**：经 `ErrorMapping`→`DomainError`，保留后端 `message`。
- **写后失效**：Repository 成功分支调用缓存失效（§18）。
- **批量部分成败**：逐项 try/catch 聚合（§17）。
- **不硬编码任何实例地址。**

### 10.2 v0.2 接入接口一览
| 功能 | 接口 | 方法 | Repo | v0.2 |
|---|---|---|---|---|
| 上传（流式） | `api/fs/put` | PUT(stream) | UploadRepository | ✅ 实现 |
| 上传（表单） | `api/fs/form` | PUT(multipart) | UploadRepository | ✅ 实现（回退/兼容） |
| 直传信息 | `api/fs/get_direct_upload_info` | POST | UploadRepository | ⏸ 仅预留 |

### 10.3 接口行为核对（✅ 已从 `openlist-ref` 源码确认 / 🔎 真机复核项）
| 项 | 结论 | 状态 |
|---|---|---|
| mkdir 请求体 | `{ "path": "<新目录全路径>" }` | ✅ `fsmanage.go MkdirOrLinkReq` |
| rename 请求体 | `{ "path", "name", "overwrite" }`；`name` **禁含 `/ \`、不得为空/`.`/`..`** | ✅ `RenameReq`+`checkRelativePath` |
| remove 是否支持批量 | 是：`{ "dir", "names":[] }`，**逐项、首错中止、无逐项结果** | ✅ `RemoveReq/FsRemove` |
| move 是否支持批量 | 是：`{ "src_dir","dst_dir","names":[],"overwrite","skip_existing","merge" }` | ✅ `MoveCopyReq/FsMove` |
| copy 是否支持批量 | 同 move（额外 `merge` 目录合并） | ✅ `FsCopy` |
| move/copy 跨存储 | **返回异步 `tasks`**；同存储可能即时完成 | ✅（"提交≠就位"，见 P7） |
| put/form 上传字段 | Header：`File-Path`(URL编码全路径)、`As-Task`、`Overwrite`、`Content-Length`/`X-File-Size`、`Content-Type`、可选 `X-File-Md5/Sha1/Sha256`、`Last-Modified`(ms)；body=流 / multipart `file` | ✅ `fsup.go` |
| 上传目标路径 | 由 `File-Path` 全路径决定（`dir=split(path)`） | ✅ |
| 同名冲突返回 | `Overwrite:false` 时 **403 "file exists"** | ✅ |
| 无权限返回 | **403 PermissionDenied**（按 `CanWriteContent/Rename/Move/Copy/Remove` 位） | ✅ `model/user.go` |
| Token 失效返回 | 401 → 复用 v0.1 `SessionManager` 失效流程 | ✅（v0.1 已验证） |
| 特殊字符路径 | 全程 `OpenListPathCodec`；上传另需 URL 编码 header | ✅（v0.1 有测试集） |
| 上传进度来源 | **服务端不回传进度**；进度=客户端已写字节/总字节 | ✅（流式模型） |
| size 未知处理 | ContentResolver 取 size；取不到用 `X-File-Size` 或 `-1` 未知流 | 🔎 真机确认驱动可接受 |
| 同存储 move/copy 是否即时 | 因驱动而异 | 🔎 真机确认，UI 兜底"提交后刷新" |
| 权限位与 `FsListResp.write` 一致性 | 位掩码=用户级，`write`=目录级 | 🔎 真机交叉验证门控 |

> 🔎 项均不阻塞工程结构与核心流程（已有 UI 兜底），在 Sprint 3–6 用自建实例核对。

## 14. Upload Implementation Plan（决策 B = WorkManager）

### 14.1 依赖新增（version catalog）
`androidx.work:work-runtime-ktx`（建议 2.9.x）、`androidx.hilt:hilt-work` + `androidx.hilt:hilt-compiler`（Hilt Worker 注入）。`:app` 关闭默认 `WorkManagerInitializer`，走 `Configuration.Provider` + `HiltWorkerFactory`。

### 14.2 组件
- **`UploadWorker`（CoroutineWorker，`@HiltWorker`）**：入参 `taskId`。取 `UploadTaskEntity` → 打开 SAF `InputStream` → 组 `File-Path`(URL编码)/`Overwrite:false`(P1)/`Content-Type`/size(`Content-Length` 或 `X-File-Size`) → 经上传 OkHttpClient `PUT api/fs/put` **流式**上传 → `ProgressRequestBody` 回调更新 `uploadedBytes` 并 `setProgress`。
- **`ProgressRequestBody`**：包装 `InputStream`，按块写 + 进度回调；**不整文件读入内存**（`v0.2_PRD §13.3`）；结束关流。
- **前台服务通知**：大文件 `setForeground`（长任务）显示进度通知（`v0.2_PRD §13.2` 允许，前台通知在 v0.2 落地基础版，复杂统一留 v0.3）。
- **取消**：`cancelWorkById` → Worker 协程取消 → 置 `CANCELLED`、关流。
- **状态回写**：Worker 各阶段更新 `UploadTaskEntity.status/uploadedBytes/errorMessage`，UI 经 Room `Flow` 观察（切页不丢状态，`v0.2_PRD §13.2`）。

### 14.3 SAF 与安全
- `ACTION_OPEN_DOCUMENT`（单/多选）+ `takePersistableUriPermission`（P6）。
- 不申请全盘存储权限；`ContentResolver` 读 DisplayName/Size/Mime；取消选择不产生任务。
- 日志脱敏：不记真实路径/URL 敏感参数/Token；失败日志脱敏（`v0.2_PRD §16.1`）。

### 14.4 完成后行为（`v0.2_PRD §13.5`）
成功→目标目录缓存失效 + (在该目录则)刷新 + `SUCCESS`；失败→`FAILED`+错误，不刷成假成功，(可选)重传；取消→`CANCELLED`，不刷新（除非已有文件成功）。

### 14.5 明确不做
断点续传/分片/秒传/多线程/目录递归上传（仅 `uploadedBytes/totalBytes` 结构预留）。

## 18. Cache Invalidation Plan

新增 `FileCacheDao` 能力（现有 `clearDirectory/deleteByInstanceId/clearAll` 之上补）：
- `deleteByPathPrefix(instanceId, pathPrefix)`：删被移动/删除目录**自身**的缓存列表（其子树）。
- 复用 `clearDirectory(instanceId, parentPath)` 失效某目录列表。

**写后失效矩阵（`v0.2_PRD §14`）**：
| 操作 | 失效 | 当前目录处理 |
|---|---|---|
| 新建目录 | 当前目录 | 重载 |
| 重命名 | 父目录 + 该路径缓存(prefix) | 重载 |
| 删除 | 当前目录 + 各删除路径(prefix) | 重载 |
| 移动 | 源目录 + 目标目录 + 被移路径(prefix) | 重载 |
| 复制 | 目标目录 | 若==当前则刷新，否则仅提示 |
| 上传 | 目标目录 | 若==当前则自动刷新 |
| 批量 | 仅成功项相关目录 | 按实际结果刷新 |

**多实例**：所有失效带 `instanceId`，**不触碰其他实例缓存**；删除实例仍走 v0.1 `deleteByInstanceId` 级联（+ v0.2 需级联清 `upload_tasks`）。**异步 task**：失效+重载后可能未即时可见 → UI "稍后刷新"提示（P7）。

|---|---|---|---|

| R4 | move/copy 异步 task："提交≠就位"(P7) | 中 | 失效+刷新+"稍后刷新"提示；真机核对同存储行为 |
| R5 | Room 转正式 Migration 出错清空用户数据 | 高 | 4→5/5→6 手写迁移 + `exportSchema` + 迁移测试(MigrationTestHelper) |
| R6 | 上传 size 未知/驱动不支持 | 中 | ContentResolver 取 size，回退 `X-File-Size`/`-1`；405→显"存储不支持上传" |
| R7 | 前台服务/通知(13+)权限与后台限制 | 中 | 申请 `POST_NOTIFICATIONS`+`FOREGROUND_SERVICE_DATA_SYNC`；切后台续传仅"尽力" |

| R9 | 特殊字符 + 上传 header 双重编码 | 中 | `OpenListPathCodec` + `File-Path` URL 编码单测 |
