# OpenList Android 原生客户端需求文档

项目代号：OpenList Native Android Client
目标平台：Android APK
推荐技术栈：Kotlin + Jetpack Compose + Retrofit/OkHttp + Room + WorkManager + Media3
目标定位：接入已有 OpenList 后端实例，实现与 Web 端功能、权限、状态、错误语义和关键交互结果的高一致性。

## 0. 需要决策的事项

| 编号   | 决策项                               | 推荐方案                                                                         | 不决策的影响                                          |
| ---- | --------------------------------- | ---------------------------------------------------------------------------- | ----------------------------------------------- |
| D-01 | 是否坚持“原生 Kotlin + Jetpack Compose” | 是。以原生为主，WebView/Custom Tabs 只作为 SSO、WebAuthn、复杂管理页兜底                         | 若改用 Flutter 或 WebView，全局体验、文件系统集成、后台任务体验会下降     |
| D-02 | 是否参考/复用官方 OpenList-Mobile         | 仅作为参考，不作为主线 fork。官方 OpenList-Mobile 定位是“把手机变成文件服务器”，而不是完整远程 OpenList 后端原生客户端 | 直接 fork 可能偏离你的目标，后续会被“本机服务端”逻辑牵制                |
| D-03 | 首版是否包含完整后台管理台                     | 推荐首版做“只读/轻量管理”，第二阶段补完整管理                                                     | 全量管理台会显著拉长周期，尤其存储驱动配置表单复杂                       |
| D-04 | “原子级对齐”范围                         | 推荐定义为：功能、权限、错误码、副作用、刷新逻辑、分享/签名行为一致；UI 不要求像素级一致                               | 若不明确，验收时容易变成无限补细节                               |
| D-05 | 是否支持多实例                           | 推荐支持。允许添加多个 OpenList 地址并切换                                                   | 不支持会限制自建实例/公网/内网多场景                             |
| D-06 | 鉴权方式                              | 必须支持账号密码/JWT、游客、管理员 Token；LDAP、2FA 首版支持；SSO/WebAuthn 走浏览器兜底                  | 只做普通登录会导致不少真实部署不可用                              |
| D-07 | 离线能力                              | 首版只做只读缓存 + 上传/下载任务队列；不做完整双向离线同步                                              | 完整离线同步复杂度高，且 OpenList 多存储后端天然不适合强同步             |
| D-08 | 上传策略                              | 必须支持普通流式上传、表单上传、后台任务上传；直传能力作为增强                                              | 大文件、弱网、不同驱动会出现大量兼容问题                            |
| D-09 | 视频播放                              | 使用 Media3 原生播放，支持 URL 刷新、字幕入口、外部播放器跳转                                        | 只用系统 Intent 播放会弱化体验；只用内置播放器会增加适配成本              |
| D-10 | 开源协议                              | 推荐 Android 客户端也按 AGPL-3.0 或兼容策略处理，避免直接复制后端代码后产生许可证风险                         | OpenList 后端/API 文档为 AGPL-3.0，前端为 MIT，复制代码边界需要谨慎 |
| D-11 | 是否做 Play 商店发布                     | 推荐先做 GitHub Release/自签 APK，稳定后再考虑商店                                          | 商店发布会增加隐私政策、权限说明、合规审查成本                         |
| D-12 | 是否允许 WebView 兜底页面                 | 推荐允许，但必须标记为“兜底能力”，不能替代主流程                                                    | 若完全禁用 WebView，SSO/WebAuthn 和复杂管理页成本会上升          |

## 1. 项目背景

OpenList 是一个支持多种存储后端的文件列表/网盘聚合项目，官方 README 列出的能力包括多存储、文件预览、图片预览、视频/音频预览、Office 预览、README 渲染、WebDAV、打包下载、Web 上传、删除、新建目录、重命名、移动、复制、离线下载、跨存储复制等 。当前主要交互入口是 Web 前端，移动端浏览器使用体验不理想，尤其在文件选择、后台上传/下载、视频播放、通知、分享、弱网恢复、长列表操作方面不如原生应用。

本项目目标是实现一个 Android 原生 APK，连接用户已有的 OpenList 后端实例，使手机端获得接近原生网盘 App 的体验，同时尽量保持与 Web 端功能语义一致。

## 2. 项目目标

### 2.1 核心目标

1. 支持连接任意用户自建 OpenList 实例。
2. 支持浏览、搜索、预览、上传、下载、分享、管理 OpenList 文件。
3. 实现与 Web 端“功能级 + 语义级”对齐。
4. 在 Android 上提供更好的文件选择、后台任务、通知、媒体播放、缓存、分享体验。
5. 后端尽量不做侵入式修改，优先复用 OpenList 已有 REST API。

### 2.2 非目标

1. 首版不做完整双向离线同步。
2. 首版不做自研云盘驱动。
3. 首版不内置 OpenList 后端服务。
4. 首版不追求与 Web 前端像素级一致。
5. 首版不做 iOS。
6. 首版不保证覆盖所有极少数驱动的特殊能力，但要有扩展接口和 Web 兜底入口。

## 3. 产品定位

本 APK 不是“把 OpenList Web 打包成 App”，而是一个真正的 OpenList 原生客户端。

推荐产品形态：

主流程：原生 Kotlin/Compose 实现。
兜底流程：SSO、WebAuthn、少数复杂管理页使用 Custom Tabs 或 WebView。
验收标准：以 OpenList Web 端行为和官方 REST API 为基准。

OpenList 后端路由中已明确存在 `/api/auth/login`、`/api/auth/login/hash`、`/api/auth/login/ldap`、`/api/me`、2FA、SSO、WebAuthn、public settings、fs、task、share、admin 等接口分组 。因此原生客户端有明确 API 基础。

## 4. 用户角色

### 4.1 普通用户

目标：在手机上方便浏览、预览、下载、上传 OpenList 文件。

核心场景：

1. 打开 App，选择一个 OpenList 实例。
2. 登录或以游客身份访问。
3. 浏览目录、搜索文件。
4. 直接播放视频、查看图片、查看 Markdown/文本/PDF。
5. 上传手机本地文件到指定目录。
6. 下载文件到手机。
7. 复制链接或分享文件。

### 4.2 管理员

目标：在手机上完成常见维护操作。

核心场景：

1. 管理用户。
2. 查看存储状态。
3. 启用/禁用存储。
4. 触发索引构建/更新。
5. 查看任务状态。
6. 修改部分全局设置。
7. 获取或重置 API Token。

OpenList 路由中已有 admin 分组，覆盖 meta、user、storage、driver、setting、task、message、index 等能力 。

### 4.3 游客用户

目标：无需登录即可访问允许公开访问的目录或分享内容。

要求：

1. 支持匿名进入实例。
2. 权限受后端控制。
3. 对无权限操作隐藏入口或提示登录。
4. 支持分享链接访问。

## 5. “原子级对齐”的定义

本项目中的“原子级对齐”不等于 UI 像素级复刻，而是指以下五类一致：

1. **功能一致**：Web 能做的核心操作，App 应具备等价入口。
2. **权限一致**：游客、普通用户、管理员看到的能力与后端权限一致。
3. **状态一致**：上传、下载、任务、分享、索引等状态与 Web/后端一致。
4. **错误一致**：401、403、429、签名失效、路径错误、驱动异常等错误应有一致语义。
5. **副作用一致**：重命名、复制、移动、删除、分享启停、索引构建等操作后刷新范围和结果一致。

验收方式：建立 Parity Matrix。每个 Web 操作都要记录“接口、请求参数、请求头、权限、成功结果、失败结果、是否刷新、是否影响缓存、Android 对应实现”。

## 6. 功能需求

### 6.1 实例管理

优先级：P0

需求：

1. 用户可以添加 OpenList 实例地址。
2. 支持 HTTP/HTTPS。
3. 支持自定义实例名称。
4. 支持多实例切换。
5. 支持测试连接。
6. 支持删除实例。
7. 支持为每个实例独立保存登录状态、缓存和任务。

验收标准：

1. 添加实例后可自动请求 `/api/public/settings` 或同等公开接口判断实例可达。
2. 多实例切换不会串 Token。
3. 删除实例时清理本地 Token、缓存、任务记录。
4. 地址错误、证书错误、网络不可达要给出明确提示。

### 6.2 登录与会话

优先级：P0

需求：

1. 支持用户名密码登录。
2. 支持 hash 登录。
3. 支持 LDAP 登录。
4. 支持 2FA OTP 输入。
5. 支持游客访问。
6. 支持管理员 Token 直接接入。
7. 支持退出登录。
8. 支持 Token 失效后重新登录。

OpenList 路由中包含普通登录、hash 登录、LDAP 登录、当前用户信息、2FA、退出登录等接口 。

验收标准：

1. 登录成功后保存 Token。
2. 所有受保护 API 自动附加 `Authorization`。
3. 401 时清理或标记会话失效。
4. 2FA 开启时弹出 OTP 输入。
5. 游客模式下不可访问的功能不展示或提示登录。
6. 管理员 Token 模式可进入管理功能。

### 6.3 SSO 与 WebAuthn

优先级：P1

需求：

1. SSO 使用 Custom Tabs 打开 OpenList 后端 SSO 页面。
2. WebAuthn 首版使用浏览器兜底。
3. 登录完成后通过回调或手动复制 Token 回到 App。
4. 原生 WebAuthn 可放到后续版本。

OpenList 路由中包含 SSO redirect/callback/token 以及 WebAuthn begin/finish login/registration 接口 。

验收标准：

1. SSO 登录失败不影响普通登录。
2. WebAuthn 页面无法原生处理时可安全打开浏览器。
3. App 内不保存第三方 SSO 密码。

### 6.4 文件列表

优先级：P0

需求：

1. 展示当前路径文件和目录。
2. 支持路径面包屑。
3. 支持返回上级。
4. 支持排序：名称、大小、修改时间、类型。
5. 支持刷新。
6. 支持列表/网格切换。
7. 支持目录树快速跳转。
8. 支持显示文件大小、修改时间、类型图标。
9. 支持 README 自动展示。

验收标准：

1. 与 Web 端同目录返回内容一致。
2. 对无权限目录给出一致错误。
3. 大目录滚动不卡顿。
4. 刷新后本地缓存更新。
5. 目录切换时路径编码正确，特殊字符不出错。

### 6.5 文件详情

优先级：P0

需求：

1. 展示文件名、路径、大小、创建/修改时间。
2. 展示类型、缩略图、签名状态、挂载信息。
3. 支持复制路径。
4. 支持复制直链/永久链接。
5. 支持打开下载链接。
6. 支持外部应用打开。

验收标准：

1. 文件详情字段尽量完整映射后端对象。
2. 下载/预览前必要时重新获取签名。
3. 签名失效时自动刷新对象信息后重试一次。

### 6.6 文件操作

优先级：P0

需求：

1. 新建目录。
2. 重命名文件/目录。
3. 批量重命名。
4. 移动。
5. 复制。
6. 删除。
7. 删除空目录。
8. 复制文件到其他存储。
9. 操作前二次确认高危动作。
10. 操作后刷新当前目录和受影响目录。

OpenList 的 fs 路由包含 put/form 上传、offline download、archive、torrent、direct upload 等接口，分享路由包含 list/get/create/update/delete/enable/disable 。

验收标准：

1. 成功后列表状态与 Web 一致。
2. 失败时显示后端错误信息。
3. 删除、覆盖、批量操作有确认弹窗。
4. 移动/复制时支持目录树选择目标路径。

### 6.7 上传

优先级：P0

需求：

1. 支持从 Android 文件选择器选择单个文件。
2. 支持多文件上传。
3. 支持上传到当前目录。
4. 支持覆盖/不覆盖选项。
5. 支持后台上传。
6. 支持上传进度通知。
7. 支持失败重试。
8. 支持大文件上传。
9. 支持 Last-Modified、hash 等元数据。
10. 支持 OpenList 直传能力，若后端/驱动可用则使用直传。

OpenList 路由中包含 `PUT /api/fs/put`、`PUT /api/fs/form`、`/api/fs/get_direct_upload_info`，并且上传路由受上传限速中间件处理 。

验收标准：

1. 小文件可稳定上传。
2. 大文件上传切后台后不中断或可恢复。
3. 上传成功后当前目录刷新。
4. 上传失败显示具体原因。
5. 覆盖策略与 Web 一致。
6. 上传任务在任务中心可见。
7. 直传失败时可回退普通上传。

### 6.8 下载

优先级：P0

需求：

1. 支持单文件下载。
2. 支持后台下载。
3. 支持系统通知进度。
4. 支持暂停/取消/重试。
5. 支持选择保存位置。
6. 支持调用系统下载管理器或自研下载器。
7. 支持分享下载链接。
8. 支持签名失效自动刷新。

OpenList 后端存在普通下载、代理、归档访问、分享下载等路由，包括 `/sd/:sid` 分享下载路径 。

验收标准：

1. 下载完成后文件可在系统文件管理器中找到。
2. 网络中断后任务状态明确。
3. 大文件下载切后台不丢任务。
4. 分享下载路径与普通下载路径区分处理。
5. 签名失效后重试一次，仍失败则提示用户刷新。

### 6.9 预览

优先级：P0/P1

需求：

P0：

1. 图片预览。
2. 视频播放。
3. 音频播放。
4. 文本预览。
5. Markdown 预览。
6. PDF 外部或内置预览。
7. Office 文件外部打开。

P1：

1. 字幕支持。
2. 歌词支持。
3. 代码高亮。
4. 编码切换。
5. 图片画廊模式。
6. README 自动渲染增强。

OpenList 官方功能列表包含 PDF、Markdown、代码、纯文本预览，图片画廊，视频/音频、歌词和字幕，Office 预览，README 渲染 。

验收标准：

1. 常见图片格式可原生查看。
2. 视频可横屏、拖动、倍速、后台控制。
3. 文本大文件采用流式或分页加载。
4. Markdown 渲染不执行危险脚本。
5. 不支持格式提供“下载/外部打开/Web 打开”。

### 6.10 搜索

优先级：P0

需求：

1. 支持当前目录搜索。
2. 支持全局搜索。
3. 支持服务端搜索。
4. 支持本地最近缓存搜索。
5. 支持搜索结果跳转目录。
6. 支持搜索空结果、索引未建立、索引异常提示。

验收标准：

1. 服务端搜索结果与 Web 一致。
2. 本地搜索明确标识为“本地缓存结果”。
3. 管理员可查看索引状态或触发索引更新。
4. 搜索失败不影响目录浏览。

### 6.11 分享

优先级：P0

需求：

1. 创建分享。
2. 查看分享列表。
3. 修改分享配置。
4. 启用/禁用分享。
5. 删除分享。
6. 复制分享链接。
7. 访问分享链接。
8. 支持分享态文件下载。

OpenList 分享路由包含 list、get、create、update、delete、enable、disable 。

验收标准：

1. 创建分享后 Web 端可见。
2. App 禁用分享后 Web 端同步不可访问。
3. 分享链接带密码/过期/权限时行为与 Web 一致。
4. 分享态下载走分享下载逻辑，不误用普通下载逻辑。

### 6.12 离线下载与任务中心

优先级：P1

需求：

1. 支持添加离线下载任务。
2. 支持查看任务列表。
3. 支持查看任务进度。
4. 支持取消任务。
5. 支持任务失败原因查看。
6. 支持管理员任务入口。

OpenList 路由中 fs 分组包含 `add_offline_download`，task 路由通过 `SetupTaskRoute` 装配，并在 admin/task 中保留兼容入口 。

验收标准：

1. 添加任务后可在任务中心看到状态。
2. 任务完成后目标目录刷新。
3. 任务失败显示后端错误。
4. 任务轮询频率可控，不造成耗电。

### 6.13 归档与压缩包

优先级：P1

需求：

1. 查看压缩包元信息。
2. 浏览压缩包内部文件。
3. 解压到指定目录。
4. 下载压缩包内部文件。
5. 分享态归档访问。

验收标准：

1. 常见压缩格式可读取或提示不支持。
2. 解压操作作为后台任务。
3. 解压完成后目标目录刷新。
4. 归档内下载路径与普通文件区分。

### 6.14 Torrent 能力

优先级：P2

需求：

1. 解析 torrent 文件。
2. 上传并解析 torrent。
3. 根据路径生成 torrent。
4. 支持快速上传能力时展示入口。

OpenList fs 路由中存在 torrent parse、upload_parse、rapid_upload、generate 等接口 。

验收标准：

1. 不支持的实例隐藏入口或显示不可用。
2. 解析结果可读。
3. 生成任务失败时返回后端错误。

### 6.15 后台管理

优先级：P1/P2

P1 必做：

1. 用户列表查看。
2. 当前用户信息修改。
3. 存储列表查看。
4. 启用/禁用存储。
5. 驱动列表查看。
6. 设置列表查看。
7. 索引构建/更新/停止/清空/进度查看。
8. 任务查看。
9. API Token 重置入口。

P2 增强：

1. 创建/编辑存储。
2. 创建/编辑用户。
3. Meta 管理。
4. 完整设置管理。
5. 驱动配置表单动态生成。
6. 消息发送/读取。

验收标准：

1. 非管理员不可进入管理台。
2. 管理操作有二次确认。
3. 高风险操作，如删除用户、删除存储、重置 Token，必须确认。
4. 创建/编辑存储若表单复杂，首版允许打开 Web 管理页兜底。

## 7. 非功能需求

### 7.1 性能

1. 冷启动到可操作不超过 3 秒。
2. 缓存命中目录首屏不超过 1 秒。
3. 远程目录加载失败不阻塞历史缓存展示。
4. 1000 文件目录滚动不卡顿。
5. 图片缩略图懒加载。
6. 视频播放不提前完整下载。

### 7.2 稳定性

1. App 崩溃后上传/下载任务可恢复或明确失败。
2. Token 失效不导致无限重试。
3. 网络错误统一进入错误处理。
4. 后端返回异常 JSON 时不崩溃。
5. 路径含空格、中文、特殊符号时正确编码。

### 7.3 安全

1. Token 使用 Android Keystore 加密保存。
2. 管理员 Token 单独加密保存。
3. 默认不保存用户密码。
4. 禁止明文日志输出 Token、Cookie、文件直链。
5. HTTPS 证书默认严格校验。
6. 自签证书支持需要用户主动开启并明确风险提示。
7. WebView 仅用于受控兜底页面，不开放任意 JS Bridge。
8. 下载文件不自动执行。
9. Markdown 不执行脚本。

### 7.4 权限

首版建议申请：

1. INTERNET。
2. POST_NOTIFICATIONS。
3. FOREGROUND_SERVICE / dataSync。
4. 通过 Storage Access Framework 选择文件，不默认申请全盘读写。
5. Android 13+ 按需申请媒体权限。

不建议首版申请：

1. MANAGE_EXTERNAL_STORAGE。
2. Camera。
3. Contacts。
4. Location。
5. SMS/Phone。

### 7.5 国际化与主题

1. 首版支持中文、英文。
2. 支持深色模式。
3. 跟随系统主题。
4. 文件大小、日期时间本地化。
5. 后端原始错误保留，同时展示本地友好提示。

## 8. 技术架构需求

### 8.1 推荐架构

采用单 Activity + Compose Navigation + MVVM + Repository + UseCase。

模块建议：

1. `app`
2. `core-common`
3. `core-network`
4. `core-database`
5. `core-auth`
6. `feature-instance`
7. `feature-files`
8. `feature-preview`
9. `feature-transfer`
10. `feature-share`
11. `feature-admin`
12. `feature-task`
13. `feature-webfallback`

### 8.2 网络层

技术：

1. Retrofit。
2. OkHttp。
3. Kotlinx Serialization 或 Moshi。
4. 统一 ApiResponse 包装。
5. 统一错误解析。
6. Authorization Interceptor。
7. Token 失效 Interceptor。
8. 下载/上传专用 OkHttpClient。

要求：

1. Base URL 按实例隔离。
2. 每个实例单独 Token。
3. 支持请求重试，但写操作默认不自动重试，除非幂等。
4. 上传/下载超时时间单独配置。
5. 支持取消请求。

### 8.3 本地数据库

使用 Room。

核心表：

1. InstanceEntity：实例信息。
2. AccountEntity：账号信息。
3. SessionEntity：Token 元信息。
4. FileCacheEntity：目录缓存。
5. FileObjectEntity：文件对象缓存。
6. TransferTaskEntity：上传/下载任务。
7. ShareEntity：分享缓存。
8. SearchHistoryEntity：搜索历史。
9. RecentPathEntity：最近访问目录。
10. AdminCacheEntity：管理台轻量缓存。

### 8.4 缓存策略

1. 目录列表缓存 TTL 默认 5 分钟。
2. 文件详情缓存 TTL 默认 5 分钟。
3. 缩略图缓存 TTL 默认 7 天。
4. README/Markdown 缓存 TTL 默认 1 天。
5. 签名下载 URL 不长期缓存。
6. 写操作成功后立即失效相关目录缓存。
7. 用户手动下拉刷新强制绕过缓存。

### 8.5 后台任务

使用 WorkManager + 前台服务通知。

任务类型：

1. 上传。
2. 下载。
3. 离线下载提交。
4. 索引构建轮询。
5. 目录刷新。
6. 缓存清理。

要求：

1. 大文件任务必须显示通知。
2. 支持取消。
3. 支持失败原因。
4. 支持仅 Wi-Fi 选项。
5. 支持低电量时暂停非关键同步。

## 9. API 对齐要求

### 9.1 API 来源

### 9.2 API 封装原则

5. 所有接口保留原始 response code 和 message。
6. 与 Web 语义不一致时，以 Web 实测行为为准。

## 14. 风险清单

| 风险                  | 等级 | 说明                    | 应对                        |
| ------------------- | -- | --------------------- | ------------------------- |
| API 文档与 Web 行为不完全一致 | 高  | 文档不一定覆盖所有边界           | 通过 HAR + Parity Matrix 验证 |
| 上传大文件兼容性            | 高  | 不同驱动行为不同              | 流式上传、任务上传、直传、重试策略并存       |
| SSO/WebAuthn 原生化复杂  | 中高 | 移动端适配成本高              | 首版使用浏览器兜底                 |
| 后台管理表单复杂            | 中高 | 驱动字段动态变化              | 首版只做常用管理，复杂表单 Web 兜底      |
| 缓存一致性               | 中  | OpenList 多存储后端存在最终一致性 | 写后失效、手动刷新、状态提示            |
| 许可证边界               | 中  | 后端 AGPL，前端 MIT        | 尽量自研，不直接复制后端代码            |
| 多实例安全隔离             | 中  | Token 串实例风险           | 数据库和会话按 instanceId 隔离     |
| 视频播放兼容性             | 中  | 各驱动直链/代理/Range 支持不同   | Media3 + 外部播放器兜底          |
| Android 后台限制        | 中  | 上传下载可能被系统杀死           | WorkManager + 前台服务通知      |
| 特殊路径编码              | 中  | 中文、空格、感叹号等路径易出错       | 建立路径编码测试集                 |

## 15. 推荐最终方案

推荐采用：

1. Kotlin + Jetpack Compose 原生客户端。
2. REST API 直接接入 OpenList 后端。
3. 以 Web 行为作为验收标准。
4. SSO/WebAuthn/复杂管理页允许 WebView 或 Custom Tabs 兜底。
5. 首版聚焦文件浏览、预览、上传、下载、分享、搜索、多实例。
6. 第二阶段补管理台、离线下载、归档、索引、任务增强。
7. 第三阶段补高级预览、原生 WebAuthn、完整管理表单、插件化能力。

## 17. 给你的最终决策建议

建议你拍板以下默认方向：

1. **做原生 Kotlin + Compose，不做 Flutter fork。**
2. **主线不复用官方 OpenList-Mobile，只参考其实现和许可证策略。**
3. **首版目标是“用户端高频功能完整”，不是“一口气复刻完整管理后台”。**
4. **允许 WebView/Custom Tabs 作为少数功能兜底。**
5. **离线能力只做缓存和任务恢复，不做完整同步盘。**
6. **验收以 Parity Matrix 为准，而不是主观感觉。**
7. **首版先通过 GitHub Release 分发 APK。**

确认这些决策后，下一步就可以进入详细交互原型、API 契约表和开发任务拆分。
