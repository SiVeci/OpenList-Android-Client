## 3. Scope Summary

v0.4 目标（`v0.4_EXECUTION_PLAN.md §2`）：

```text
v0.4 = 统一文件打开分发 + 图片预览 + 文本预览 + Markdown 预览
     + PDF/Office 外部打开 + 视频播放器 + 音频播放器 + 基础字幕支持
     + 不支持格式兜底
```

**必做 20 项**（`v0.4_PRD.md §5.1`）：统一文件打开分发；文件列表/文件详情/搜索结果/分享详情/任务完成结果 5 个预览入口；图片预览；文本预览；Markdown 预览；PDF 外部打开；Office 外部打开；视频播放器；音频播放器；基础字幕支持；不支持格式兜底；预览缓存；预览错误处理；Token 失效和签名 URL 失效处理；多实例隔离；v0.1/v0.2/v0.3 回归验证。

**明确不做**：PDF/Office 内置预览、代码高亮、歌词、图片画廊、归档浏览、Torrent 预览、完整分享态目录浏览、任意分享 URL 入站解析、视频转码、字幕在线搜索/下载/时间轴编辑、自研下载器、管理台、v1.0 原子级对齐（详见 `v0.4_PRD.md §5.3` 与 `KNOWN_ISSUES.md`）。

**本版本额外明确裁剪一项**（S3 实现阶段判断）：Markdown 内嵌图片渲染——需要额外的 `markwon-image` 依赖 + 自研 Coil `AsyncDrawableLoader`，评估后判断超出本版本合理工作量，留待后续版本。

## 5. Implemented Scope

### 5.1 模块架构演进

Status: **Accepted**

Evidence:
- 新增 `:feature:preview` Gradle 模块（`settings.gradle.kts`），依赖方向 `feature:preview → core:{common,model,domain,designsystem}`，不依赖任何其他 feature 模块（S5 期间一度出现依赖越界，已发现并重构修正，见 §4 表格）。
- `:app` 内 `feature/preview/Reserved.kt` 占位包已删除并迁移为真实模块；`feature/admin`、`feature/webfallback` 占位保留至 v0.5（按计划）。

### 5.2 Room 数据库迁移

Status: **Accepted（手写 SQL + 逐字段核对通过；未经 KSP 实际导出验证，真机升级路径未验证）**

Evidence:
- `OpenListDatabase` 升级到 version 8，新增 `PreviewCacheEntity`（`id/instanceId/path/kind/mimeType/lastModified/cacheKey/localFilePath/sizeBytes/etag/expiresAt/cachedAt` + `(instanceId,path,kind)` 唯一索引）。
- `MIGRATION_7_8` 手写 SQL 与手工编写的 `schemas/.../8.json` 逐字段核对一致（三方核对表见 `v0.4_IMPLEMENTATION_LOG.md` S1 章节）。
- **与 v0.1-v0.3 迁移的关键差异**：本轮无法运行 Gradle/KSP，`8.json` 的 `identityHash` 是手工填的占位值，不是真实导出值——这是本版本相比历史版本更弱的一环，下次进入可构建环境时必须重新用 KSP 实际导出并核对，这是发布前的强制前置步骤，不是可选项。
- `InstanceRepositoryImpl.delete` 级联清理扩展到 `preview_cache` 表 + 对应磁盘缓存目录（`cacheDir/preview/<instanceId>/`）。

### 5.3 四个新 Repository

Status: **Accepted**

Evidence:
- `PreviewRepository`/`MediaRepository`/`SubtitleRepository`/`ExternalOpenRepository` 接口位于 `:core:domain`，实现位于 `:data:repository`，均已接入 Hilt `RepositoryModule`。
- 均遵循与 v0.1-v0.3 一致的 instance 查找 → `instanceContext.set` → `clientFactory.apiFor` → `safeApiCall{ api.fsGet(...) }` → 401 → `sessionManager.invalidate` 模式，未各自发明新的错误处理方式（S7-T2 权限矩阵核查确认）。
- `FileOperationRepositoryImpl`/`UploadWorker` 新增对 `PreviewRepository.invalidate`/`invalidateByPrefix` 的调用——**本仓库首次出现 Repository 依赖 Repository**，明确注释为"仅限写操作成功后缓存失效副作用场景的特例，不是可效仿的通用模式"。

### 5.4 后端接口对照（V-401~V-410）

Status: **Accepted（provisional，部分依赖不在本地 `openlist-ref/` checkout 范围内）**

Evidence（详见 `v0.4_IMPLEMENTATION_LOG.md` S0 章节）：
- `raw_url`/`/d/`/`/p/` 优先级、鉴权失败响应形态（HTTP 状态码+JSON body）、字幕候选可靠性（`fs/get` 的 `related` 字段）、分享路径预览可行性（创建者原始路径可走普通 fs/get）——均已对照后端源码在 handler/路由层核实。
- 签名有效期、Range 转发内部实现、签名验证中间件具体实现——不在本地精简 checkout 范围内，标注为 provisional，采用防御式策略（每次重新 fs/get 不复用旧 URL；不假设 Range 一定可用；4xx 统一触发一次刷新不区分具体原因）。

### v1.0 Parity 差异记录（PRD §19.2 八项，摘要）

1. 图片预览：与 Web 端行为差异未系统对比（Web 端具体交互未知）。
2. 文本预览：512KB/20MB 二级截断策略是客户端自定义，未对照 Web 端阈值。
3. Markdown 渲染：原生渲染 vs Web 端可能的富文本引擎，排版细节必然存在差异；无内嵌图片是本版本明确的功能差距。
4. PDF/Office：外部打开 vs Web 端可能的在线预览，能力差距明确（架构决策 D-02/D-03）。
5. 视频播放兼容性：依赖 Media3/ExoPlayer 默认解码器支持面，与 Web 端浏览器原生解码能力/驱动直链支持面必然不同。
6. 音频播放兼容性：同上。
7. 字幕支持范围：仅同目录发现+手动选择，无在线字幕库对接，与更完整的 Web 端（若有）能力有差距。
8. 不支持格式兜底：客户端统一走外部打开/下载/Web 打开三选一，Web 端可能有自己的预览器 fallback 逻辑，未对比。
