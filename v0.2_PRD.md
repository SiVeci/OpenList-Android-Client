# OpenList Android 原生客户端 v0.2 需求文档

版本：v0.2
阶段定位：上传与文件操作闭环版
前置版本：v0.1 已完成开发、执行与验收
目标平台：Android
推荐技术栈延续：Kotlin + Jetpack Compose + Retrofit/OkHttp + Room + DataStore + WorkManager
正式引入设计约束：根目录 `DESIGN.md`
核心目标：在 v0.1 已完成的“多实例 + 登录/游客 + 文件浏览 + 文件详情 + 基础下载”基础上，补齐上传、文件写操作、批量选择、目录选择器、缓存一致性和基础设计系统。

---

## 1. 版本背景

v0.1 已完成第一个最小纵向闭环：

```text
添加实例
登录 / 游客访问
进入文件列表
浏览目录
查看文件详情
基础下载
```

v0.2 的目标是在这个基础上，让客户端从“只能浏览和下载”升级为“可以在手机上管理 OpenList 文件”。

因此，v0.2 不追求一次性实现所有长期需求，而是聚焦移动端最核心的文件管理能力：

```text
上传
新建目录
重命名
删除
移动
复制
批量选择
目录选择器
写操作后缓存刷新
基础设计系统落地
```

## 2. 前序要求总结

### 2.1 长期产品目标

OpenList Android 原生客户端的长期目标是实现一个真正的原生 APK，而不是 WebView 包壳。

长期方向包括：

1. 多实例管理。
2. 登录、游客访问、管理员访问。
3. 文件浏览。
4. 文件详情。
5. 上传。
6. 下载。
7. 搜索。
8. 分享。
9. 任务中心。
10. 图片、文本、Markdown、PDF、Office 等预览。
11. 视频/音频播放。
12. 离线下载。
13. 归档能力。
14. 轻量管理台。
15. 与 Web 端功能、权限、错误语义、状态、副作用进行原子级对齐。

### 2.2 已确认的开发节奏

项目采用渐进式开发：

```text
v0.1：技术验证与最小闭环
v0.2：上传与文件操作
v0.3：分享与任务中心
v0.4：预览与播放器
v0.5：轻量管理台
v1.0：原子级对齐稳定版
```

开发原则：

1. 先跑通纵向闭环。
2. 再逐步填充功能模块。
3. 不提前扩大当前版本范围。
4. 不为了长期完整性牺牲当前版本可交付性。
5. 每个版本都要为后续版本预留架构位置。

### 2.3 v0.1 默认已完成能力

v0.2 可以假设 v0.1 已经具备：

1. 添加实例。
2. 删除实例。
3. 切换实例。
4. 测试连接。
5. 普通登录。
6. 游客访问。
7. Token 保存。
8. 当前用户信息获取。
9. 根目录浏览。
10. 子目录浏览。
11. 文件详情。
12. 基础下载。
13. 目录缓存。
14. 多实例隔离。
15. Token 失效处理。
16. 基础设置页。
17. 日志脱敏。
18. APK 构建。

如果实际仓库与上述能力不一致，coding agent 必须在制定执行计划时实时向用户汇报。

## 3. v0.2 版本定位

v0.2 是“上传与文件操作闭环版”。

一句话定义：

```text
v0.2 = 上传 + 文件写操作 + 批量选择 + 目录选择器 + 缓存一致性 + 基础设计系统
```

v0.2 的用户价值是：

1. 用户可以从手机上传文件到 OpenList。
2. 用户可以在手机上完成常见文件管理操作。
3. 用户可以批量处理文件。
4. 用户可以选择目标目录进行移动、复制和上传。
5. 用户可以获得比移动 Web 更好的原生交互体验。
6. v0.3 的分享和任务中心可以自然接入 v0.2 的文件操作与任务模型。

## 4. v0.2 需要决策的事项

本章节列出当前推荐决策。coding agent 在执行计划或实现过程中如果遇到新的关键决策，必须实时向用户汇报，不允许堆到最后。

### D-01：v0.2 是否必须包含上传？

推荐：必须包含。

原因：

上传是 Android 原生客户端相对移动 Web 的核心价值之一。如果 v0.2 不做上传，版本价值提升有限。

影响：

1. 需要接入 Android 文件选择器。
2. 需要实现上传 Repository。
3. 需要设计上传任务数据结构。
4. 需要上传进度展示。
5. 需要为后续任务中心预留基础。

---

### D-02：v0.2 是否做完整任务中心？

推荐：不做完整任务中心。

v0.2 做：

1. 上传任务基础状态。
2. 上传进度展示。
3. 上传取消。
4. 上传失败提示。
5. 简单上传记录或当前上传状态。

v0.2 不做：

1. 完整任务中心页面。
2. 后端任务统一轮询。
3. 下载、上传、离线下载、索引任务统一管理。
4. 任务历史复杂筛选。

完整任务中心放到 v0.3。

---

### D-03：v0.2 是否做断点续传？

推荐：不做完整断点续传，只做结构预留。

v0.2 做：

1. 流式上传。
2. 进度统计。
3. 取消上传。
4. 失败后重新上传。
5. 数据结构预留 `uploadedBytes`、`totalBytes`。

v0.2 不做：

1. 分片上传。
2. 秒传。
3. 真正断点续传。
4. 多线程上传。

---

### D-04：v0.2 是否支持目录上传？

推荐：不支持完整目录上传。

v0.2 做：

1. 单文件上传。
2. 多文件上传。
3. 上传到当前目录。
4. 上传到指定目录。

v0.2 不做：

1. 本地目录递归上传。
2. 保留本地目录结构上传。
3. 文件夹批量导入。

---

### D-05：删除是否做客户端回收站？

推荐：不做客户端回收站。

原则：

1. 删除语义完全以后端为准。
2. 客户端只做确认和结果展示。
3. 不在客户端模拟回收站。
4. 不做撤销删除，除非后端明确支持。

---

### D-06：移动/复制是否支持跨存储？

推荐：不在客户端人为限制。

要求：

1. 允许用户选择任意可访问目录作为目标目录。
2. 是否支持跨存储，由后端返回结果决定。
3. 客户端只负责展示成功或失败。
4. 失败原因展示后端 message。

---

### D-07：v0.2 是否加入搜索？

推荐：不加入。

原因：

v0.2 聚焦上传和写操作。搜索放到后续版本，避免范围扩大。

---

### D-08：v0.2 是否正式引入 `DESIGN.md`？

推荐：正式引入。

原因：

v0.2 会新增大量强交互 UI，包括：

1. 上传入口。
2. 操作菜单。
3. 新建目录弹窗。
4. 重命名弹窗。
5. 删除确认弹窗。
6. 目录选择器。
7. 批量选择栏。
8. 上传进度。
9. 错误态。
10. 空状态。

如果此时不引入设计系统，后续 UI 统一成本会很高。

## 7. 用户场景

### 7.1 上传手机文件到 OpenList

用户在文件列表页点击上传按钮，选择手机本地文件并上传到当前目录。

流程：

```text
进入文件列表页
点击上传
选择一个或多个本地文件
确认目标目录
开始上传
显示上传进度
上传完成
刷新目录
```

验收标准：

1. 单文件上传成功。
2. 多文件上传成功。
3. 上传过程中 UI 不阻塞。
4. 上传失败有明确提示。
5. 上传完成后目标目录刷新。
6. 文件出现在目标目录中。
7. Token 失效时返回登录流程。

---

### 7.2 新建目录

用户在当前目录点击“新建目录”，输入目录名称后创建。

验收标准：

1. 目录名不能为空。
2. 目录名不能是纯空格。
3. 创建成功后刷新当前目录。
4. 重名或无权限错误展示后端 message。
5. 游客态不显示入口或提示登录。

---

### 7.3 重命名文件或目录

用户从文件操作菜单选择重命名。

验收标准：

1. 输入框默认填入当前名称。
2. 空名称禁止提交。
3. 成功后刷新父目录。
4. 失败后保留原状态。
5. 后端错误清晰展示。

---

### 7.4 删除文件或目录

用户选择单个或多个项目后删除。

验收标准：

1. 删除前必须二次确认。
2. 删除目录时明确提示可能删除目录及其内容。
3. 批量删除时显示项目数量。
4. 取消删除不会调用 API。
5. 成功后刷新当前目录。
6. 失败项需要展示原因。

---

### 7.5 移动文件或目录

用户选择文件或目录，打开目录选择器，选择目标目录后执行移动。

验收标准：

1. 目录选择器可用。
2. 可以进入子目录。
3. 可以返回上级。
4. 可以选择当前目录。
5. 移动成功后源目录和目标目录缓存失效。
6. 当前目录刷新。
7. 跨存储失败时展示后端错误。

---

### 7.6 复制文件或目录

用户选择文件或目录，选择目标目录后复制。

验收标准：

1. 操作流程与移动一致。
2. 复制成功后目标目录缓存失效。
3. 如果当前目录就是目标目录，必须刷新。
4. 失败时展示后端错误。
5. 批量复制支持部分失败结果。

---

### 7.7 批量选择与批量操作

用户长按文件项进入选择模式，可选择多个文件或目录。

v0.2 支持：

1. 批量删除。
2. 批量移动。
3. 批量复制。

v0.2 不支持：

1. 批量重命名。
2. 批量分享。
3. 批量下载增强。
4. 批量压缩。
5. 批量预览。

## 8. 权限与状态规则

### 8.1 游客态

游客态默认：

1. 不显示上传按钮。
2. 不显示新建目录按钮。
3. 不显示重命名入口。
4. 不显示删除入口。
5. 不显示移动入口。
6. 不显示复制入口，除非后端明确允许。
7. 用户通过其他路径触发写操作时，引导登录。

### 8.2 普通登录用户

普通用户是否能操作，以后端权限为准。

客户端规则：

1. 不硬编码用户是否能写。
2. 优先根据后端返回的权限或能力字段判断。
3. 如果无法提前判断，可展示入口，但失败时展示无权限。
4. 后端返回 403 时展示“无权限执行该操作”。

### 8.3 管理员 Token 用户

管理员 Token 用户默认展示 v0.2 文件操作入口，但最终仍以后端 API 结果为准。

### 8.4 Token 失效

写操作中如果返回 401：

1. 停止当前操作。
2. 标记当前实例会话失效。
3. 引导用户重新登录。
4. 不影响其他实例。
5. 不把失败操作标记为成功。

## 9. API 需求

### 9.1 v0.2 新增或正式启用的接口

在 v0.1 API 基础上，v0.2 需要新增或正式启用：

| 功能   | 接口                               | 说明             |
| ---- | -------------------------------- | -------------- |
| 上传   | `/api/fs/put`                    | 普通上传           |
| 表单上传 | `/api/fs/form`                   | 兼容表单上传方式       |
| 直传信息 | `/api/fs/get_direct_upload_info` | v0.2 只预留，不强制使用 |
| 新建目录 | `/api/fs/mkdir`                  | 创建目录           |
| 重命名  | `/api/fs/rename`                 | 文件或目录重命名       |
| 移动   | `/api/fs/move`                   | 文件或目录移动        |
| 复制   | `/api/fs/copy`                   | 文件或目录复制        |
| 删除   | `/api/fs/remove`                 | 文件或目录删除        |
| 文件列表 | `/api/fs/list`                   | 操作后刷新          |
| 文件详情 | `/api/fs/get`                    | 操作前后校验或详情刷新    |

### 9.2 API 接入原则

1. 所有写操作必须通过 Repository。
2. UI 层禁止直接调用 Retrofit。
3. 所有写操作必须统一错误处理。
4. 所有路径参数必须经过统一路径编码层。
5. 所有写操作必须携带 `instanceId`。
6. Token 由 Interceptor 自动注入。
7. 写操作成功后必须触发缓存失效。
8. 批量操作必须支持部分成功、部分失败。
9. 后端 message 必须尽量保留并展示给用户。
10. 不得硬编码具体测试实例地址。

### 9.3 接口行为验证要求

coding agent 在执行计划中需要列出接口验证项：

1. mkdir 的请求体格式。
2. rename 的请求体格式。

## 13. 上传技术要求

### 13.1 Android 文件选择

使用 Android Storage Access Framework。

要求：

1. 不申请全盘存储权限。
2. 支持单选。
3. 支持多选。
4. 使用 `ContentResolver` 读取文件流。
5. 不将文件完整读入内存。
6. 尽量读取 DisplayName、Size、MimeType。
7. 文件选择取消时不产生错误任务。

---

### 13.2 上传执行方式

v0.2 推荐实现：

1. 小文件可以直接协程上传。
2. 较大文件通过 WorkManager 或可取消协程任务执行。
3. 上传状态写入本地数据库。
4. 上传过程中切换页面不丢状态。
5. 上传完成后刷新目标目录。
6. 切后台继续上传作为尽力支持，不作为 v0.2 强验收。
7. 前台服务通知可预留到 v0.3 任务中心统一实现。

---

### 13.3 上传流式处理

要求：

1. 不一次性读取完整文件到内存。
2. 使用 InputStream 流式上传。
3. 支持进度回调。
4. 支持取消。
5. 关闭输入流。
6. 网络失败时正确标记任务失败。

---

### 13.4 上传进度

要求：

1. 每个文件显示独立进度。
2. 多文件上传显示整体状态。
3. 上传失败显示原因。
4. 上传成功显示完成状态。
5. 用户可取消未完成任务。
6. 上传状态可被后续任务中心复用。

---

### 13.5 上传完成后行为

上传成功后：

1. 目标目录缓存失效。
2. 如果用户当前就在目标目录，自动刷新。
3. 显示成功提示。
4. 上传记录状态变为 `SUCCESS`。

上传失败后：

1. 上传记录状态变为 `FAILED`。
2. 显示错误信息。
3. 支持重新发起上传，可选。
4. 不刷新成本地假成功状态。

上传取消后：

1. 上传记录状态变为 `CANCELLED`。
2. 不刷新目标目录，除非部分文件已经成功上传。
3. 已完成文件需要保留成功状态。

## 14. 写操作缓存一致性要求

v0.2 必须重点处理缓存一致性。

### 14.1 新建目录

成功后：

1. 当前目录缓存失效。
2. 当前目录重新加载。

### 14.2 重命名

成功后：

1. 父目录缓存失效。
2. 文件详情缓存失效。
3. 当前目录重新加载。

### 14.3 删除

成功后：

1. 当前目录缓存失效。
2. 被删除路径缓存删除。
3. 当前目录重新加载。

### 14.4 移动

成功后：

1. 源目录缓存失效。
2. 目标目录缓存失效。
3. 被移动路径旧缓存删除。
4. 当前目录重新加载。

### 14.5 复制

成功后：

1. 目标目录缓存失效。
2. 如果当前目录是目标目录，必须刷新。
3. 如果当前目录不是目标目录，可以只提示成功。

### 14.6 上传

成功后：

1. 上传目标目录缓存失效。
2. 如果用户当前就在目标目录，则自动刷新。

### 14.7 批量操作

批量操作后：

1. 成功项相关缓存失效。
2. 失败项不应被本地删除或修改。
3. UI 展示部分成功结果。
4. 当前目录按实际结果刷新。

## 16. 安全要求

1. 游客态隐藏写操作。
2. 401 触发重新登录。
3. 403 显示权限不足。
4. 管理员 Token 只在当前实例生效。
