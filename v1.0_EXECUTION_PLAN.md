## 0. 环境与口径前言（实现阶段必读）

以下事实来自用户提供的环境说明与本轮仓库核实，是全计划的执行口径基础：

1. **构建口径**：实现机默认 `JAVA_HOME` 指向 JDK 10（过旧），全部 Gradle 命令必须显式使用 `JAVA_HOME="C:/Program Files/Java/jdk-20" ./gradlew ...`。v0.5 已在该环境实测 `assembleDebug`/`testDebugUnitTest` 通过，v1.0 沿用并强化：**每个 Sprint 出口必须实测 `compileDebugKotlin` + `testDebugUnitTest` 通过并按 Sprint 提交，禁止沿用 v0.5 执行计划早期草案的纯静态审查口径**。若实现环境的 JDK 路径变化，以实际可用 JDK 17+ 为准并在 `v1.0_IMPLEMENTATION_LOG.md` 记录。
2. **设备口径**：本轮有可用 AVD `Medium_Phone`（API 36，Play 镜像，`/data` 磁盘紧张）与用户真机。`KNOWN_ISSUES.md` 中「无真机/模拟器」的说法**已过时**，v1.0 各真机项以此设备组合验证。PRD §13.2 要求的 Android 10/API 29 目前无对应设备 → 列为需要决策事项 DEC-605。
3. **真实实例**：`https://nas.siveci.com:18443`（openresty 反向代理部署）。它既是 API 行为验证的第二依据，也是两个未入库问题（§4.3）的复现环境；回归判定时必须区分「App 缺陷」与「该实例反代配置问题」（§15 风险 R11）。
4. **后端参考源码**：本轮核实 **仓库中不存在 `openlist-ref/` 目录**（`.gitignore:38` 显式忽略，v0.5 的 checkout 未入库且已不在工作区）。用户环境说明中「openlist-ref 为完整后端源码 checkout」与仓库现状不符，**以仓库为准**：S0 第一项任务即重新获取完整后端源码 checkout（含 `server/handles/` 下 `ldap_login.go`、`auth.go`、`task.go`、`sharing.go`、`fsread.go`，及 `server/router.go`、`server/static/static.go`），仍不入库。获取后它是 v1.0 全部后端行为验证的**第一依据**，真实实例为第二依据；凡两依据均覆盖到的结论**不得默认标 provisional**。
5. **版本基线**：`versionCode=5`（`app/build.gradle.kts:35`）、`appVersionName="0.5.0"`（`gradle/libs.versions.toml:2`）；Room version=9（`OpenListDatabase.kt:41`），`9.json` identityHash 为 KSP 核对后的真实值，`8.json` 占位符保留（见 §4.2）；`Parity_Matrix.md` 不存在（本计划安排其模板定稿与填充时机，见 §11 S0/S6）。
6. 本计划中用户提供的后端「初步抽查结论」一律标注为**待 S0 复核**，S0 复核通过后才可写入实现代码注释与 Parity Matrix 作为验证结论。

## 2. v1.0 范围确认

以 `v1.0_PRD.md` §18 为准：

```text
v1.0 = Parity Matrix + 真机/模拟器回归 + UI/安全/缓存/错误统一
     + 登录补齐（LDAP / 2FA-OTP）
     + 目录级权限门控（fs/list write + 会话权限位 + 后端 403 兜底）
     + 上传失败一键重试 + 本地下载取消
     + 分享链接入站最小闭环
     + Markdown 内嵌图片最小支持
     + 管理台索引路径选择 + 管理任务批量轻操作
     + v1.0.0 稳定发布准备（versionCode=6 / versionName="1.0.0"）
```

范围裁决规则：任何补齐项若在 S0 验证后发现「API 不可确认或行为与假设冲突」，按 PRD §4.2.D.7 的先例**降级为 Web/外部浏览器兜底并记入 Parity Matrix**，不得为保住功能而猜测实现（各项降级策略见 §6、§15）。

可选范围（PRD §4.3 的 7 项：搜索分页/类型筛选、分享列表筛选、管理台搜索/筛选增强、最近预览、验收模板优化）**不进入本计划的 Sprint 任务表**，仅当对应 Sprint 提前完成且不引入新验证面时可做，一律不进强验收。

## 3. Out of Scope（明确不做）

以 `v1.0_PRD.md` §4.4（19 项）为唯一清单，此处不复述。执行时重点防越界的高风险项：

- 完整 Web 管理台 CRUD（用户/存储/Meta/消息/扫描）、动态驱动配置表单；
- 完整设置编辑/保存/删除、API Token 原生重置；
- 分享态上传、分享态完整目录写操作、分享统计/审计/二维码；
- 完整离线同步、自研下载器、断点续传、分片/多线程上传下载；
- PDF/Office 内置预览、代码高亮、歌词、图片画廊、归档浏览；
- SSO/WebAuthn 原生化（仅提供浏览器兜底说明 + Token 手动回退路径，PRD §4.2.B.4）；
- Play 商店发布、iOS。

补充边界（本计划明确）：分享入站的「分享态目录完整浏览」不做——入站最小闭环止于「解析 → 展示分享基础信息 → 单文件预览/外部打开分发 → 复制/浏览器兜底」（PRD §4.2.D.3/D.4）；`/@s/` 前缀下的多级目录递归浏览记 Deferred 入 Parity Matrix。

### 4.3 两个已复现、未入库的问题（基线必须覆盖）

以下两问题已在真机/模拟器 + `nas.siveci.com:18443` 复现，但未写入任何仓库文档，本计划正式纳入：

1. **反代环境 `fs/get` 返回的 `raw_url` 丢端口**：根因是服务端 `site_url` 配置（openresty 反代下服务端拼 raw_url 时未带 `:18443`），导致预览/下载 URL 指向错误端口。**这是服务端配置问题，不是 App 缺陷**；App 侧是否加固（host 相同、端口不同时重写为实例端口 + 预览响应 content-type 粗校验）→ **DEC-607**。无论决策结果如何，S0 须在真实实例复现并记录（V-612），S6 回归时该实例的预览/下载失败必须先对照此问题归因。
2. **`ErrorMapping.toDomainError()` 无 `HttpException` 分支**：反代返回裸 HTTP 错误页（400/502 等非信封 JSON 响应）时，Retrofit suspend 调用抛 `HttpException`，被映射为 `Unknown` → UI 只显示「未知错误」。已复核代码属实（§4.1 B13）。**修复无架构争议，直接纳入 S1**（新增 HttpException 分支：按 `code()` 走既有 `codeToDomainError` 语义 + 保留可读 message），并补单测覆盖 400/401/403/404/5xx 裸响应。

## 5. 架构增量

v1.0 **不新增大模块**，全部补齐落在既有模块内：

## 10. 缓存、刷新、安全、权限、错误处理计划

### 10.1 缓存与刷新

- **目录权限随 `fs/list` 刷新**：`DirectoryCapability` 只存在于 `FileListResult` 内存流，不建长期缓存不入库；缓存命中（Cached）阶段能力=unknown，Fresh 到达后覆盖。
- **上传重试**：成功后精确失效目标目录缓存（复用既有写操作失效路径）；重试只更新原任务行。
- **下载取消**：状态置 CANCELLED；与 DownloadManager 查询不一致时以系统状态刷新结果为准（V-606）。
- **分享入站不污染普通目录缓存**：`getInboundShare` 结果一律内存态，不写 `file_cache`；仅当（PRD §10.1.3）确认映射到普通路径且当前会话有权限、并实际走了普通 `fs/*` 通道时才允许普通缓存介入。
- **Markdown 图片**：失败不影响 Markdown 正文缓存；图片缓存 key 带 `instanceId`；文件写操作联动失效沿用 v0.4 预览缓存策略。
- **管理批量操作后**：强制刷新对应类型 undone/done 内存态，不入库；索引路径更新成功后刷新进度、不假设立刻完成（沿用 v0.5 异步语义提示）。

### 10.2 安全与脱敏

- Token（Keystore）、OTP（内存态、不入库不入日志）、分享密码（仅请求参数）、签名 URL/Authorization/Cookie/password/secret/key/sign（`SafeLogger` 脱敏面复核扩展到新增代码路径）。
- Markdown 外部图片与外部链接**不注入 Token/Authorization**（loader 层结构性保证：仅同实例 host 且走签名 URL 家族的请求可复用既有 headers 机制，其余零 header）。
- 分享入站仅自动处理已配置实例同源链接；跨域/未配置实例提示添加实例或浏览器打开；**不从分享 URL 提取保存 Token**。
- Web 兜底继续零 Token/Cookie 注入；管理批量操作二次确认 + 影响范围展示。
- S7 安全审计沿用 v0.5 双独立审计范式（指针：`v0.5_ACCEPTANCE_REPORT.md` §7），覆盖面加上本轮新路径（OTP、分享密码、图片 loader、批量操作）。

### 10.3 权限门控组合（PRD §4.2.B.5）

判定链：目录级 `write`（fs/list） ∧ 会话权限位（`Session.canDo`） → 入口展示；后端 403 → 最终裁决提示。任何一环不可得退化为下一环，最坏退回 v0.2 现状（乐观展示）。游客既有隐藏逻辑不回退。

### 10.4 错误处理

- **S1 修复 `HttpException` 分支**（§4.3.2）：按 HTTP status 走 `codeToDomainError` 语义；5xx 裸响应映射 ServerError，400 等保留 code+可读文案，不再落 Unknown。
- 新增/复用 `DomainError` 子类（PRD §12.1 全九项）：`AuthMethodUnavailable`、`OtpRequired`、`OtpInvalid`、`DirectoryPermissionUnknown`（评估是否需要独立子类或仅 UI 态）、`UploadRetryUnavailable`、`DownloadCancelUnavailable`、`ShareLinkUnsupported`、`SharePasswordRequired`、`AdminBatchOperationFailed`（仅 DEC-603 选 some 系时需要逐项载荷）。每项配 `toUserMessage` 可操作文案（重试/刷新/浏览器打开/复制/返回）。
- 401/402/403 分层：401 → 既有会话失效链路；信封 402 → OTP 流程（仅登录上下文）；403 → 权限文案不伪装空态。
- 后端 message 可信时优先展示（操作类失败经 `OpenListError` 透传，沿用 v0.5 先例）。

## 16. 需要决策事项（DEC-601 ~ DEC-607）

以下决策在 S0 结束前必须按 `AGENTS.md` 实时决策格式逐项向用户汇报关闭；每项已给推荐方案。

**DEC-601：2FA/OTP 的 UI 形态**
- A. 登录页内联第二步（同页状态机：提交→NeedOtp→展示 OTP 输入区→二次提交）
- B. 独立 OTP 对话框
- 推荐 **A**：状态与首段凭据同 ViewModel 持有，避免对话框在进程重建/旋转时丢态；与登录页四方式区分的布局改造一次做完。

**DEC-602：分享入站来源范围**
- 背景：PRD §4.2.D.2 要求 ACTION_VIEW + App 内解析两类来源；但用户实例 host 完全自配置，静态 intent-filter 无法窄声明——scheme-only 声明会让本 App 出现在**所有** https 链接的打开候选中（Android 12+ 非验证链接还默认不弹候选），既扰民又无法通过 App Links 验证。
- A. v1.0 只做 App 内粘贴输入 + 进入分享页时剪贴板检测提示，不注册 ACTION_VIEW；PRD 的 ACTION_VIEW 项记 Parity「Deferred（平台机制限制）」
- B. 仍注册 scheme=https 的 ACTION_VIEW（承受上述副作用）
- 推荐 **A**：这是对 PRD 必做项的范围修正，属必须用户裁决的冲突项；A 保住「入站最小闭环」核心价值且零平台副作用。

**DEC-603：管理任务批量接口落地子集**
- A. 只做 `clear_done`/`clear_succeeded`/`retry_failed`（无请求体三端点，恰为 PRD §4.2.F.3 点名范围：清理已完成/已成功 + 重试失败）
- B. 全部 6 个（含 some 系逐项选择，需多选 UI + `AdminTaskBatchResult` 聚合）
- 推荐 **A**：B 超出 PRD 点名范围且响应载荷待复核；some 系记 Parity Deferred。

**DEC-604：索引 `maxDepth` 默认值**
- A. V-609 实测确认 `-1` 可靠则保留，UI 文案「不限深度」
- B. 实测不可靠则改显式大值（与后端 `MaxIndexDepth` 默认 20 同源语义），UI 注明
- 推荐 **A 优先、以实测定 B**：本项本质由 V-609 事实驱动，S0 后自动收敛，无实测结果前不改代码。

**DEC-605：v1.0 验收口径与 API 29 覆盖**
- A. 验收必须真机/模拟器通过才标 Accepted（PRD §13.2 明确不得纯静态）；API 29 以「新建 API 29 AVD（磁盘允许时）」补覆盖
- B. 同上，但磁盘不允许时 API 29 记 KNOWN_ISSUES + Parity 差异，validate minSdk 静态面（编译/desugaring/API 门槛审查）替代
- 推荐 **B**：真机/模拟器通过是硬门槛不让步；API 29 物理缺失时降级为受控已知限制，不阻塞 v1.0.0，后续版本补验。

**DEC-606：Markdown 内嵌图片依赖选型**
- A. `markwon-image`（4.6.2 同版本）+ 自写受限 SchemeHandler/loader（可桥接 Coil 亦可独立 OkHttp 受限加载）
- B. `markwon-image-coil`（与项目 Coil 2.7 的编译兼容性存疑，其发布期对应 Coil 1.x/2.x 早期）
- C. 不引依赖，仅支持占位（放弃必做项，需改 PRD）
- 推荐 **A**：KDoc（§4.1 B10）本就预设此路线；受限 loader 自写才能结构性保证零凭据注入与 instanceId 缓存隔离；S4-T1 编译实测后若 A 亦不可行再实时汇报。

**DEC-607：raw_url 丢端口的 App 侧加固是否纳入 v1.0**
- A. 纳入最小加固：`fs/get` 响应 raw_url 的 host 与实例相同而端口不同时重写为实例端口；预览/播放响应 content-type 粗校验（text/html 疑似错误页时提前失败给出可读文案）。明确标注这是对服务端 `site_url` 配置问题的容错。
- B. 不加固：仅在 KNOWN_ISSUES/Parity 记录「需正确配置服务端 site_url」，回归时以修正配置后的实例为准。
- 推荐 **A**：改动小（一处 URL 后处理 + 一处响应头检查）、可单测、显著降低反代部署用户的失败率；但需用户确认接受「客户端补偿服务端配置」的原则性取舍。
