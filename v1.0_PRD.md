# OpenList Android 原生客户端 v1.0 需求文档

版本：v1.0
阶段定位：功能补齐 + 原子级对齐稳定版
前置版本：v0.1、v0.2、v0.3、v0.4、v0.5 已完成开发、执行与验收
目标平台：Android APK
核心目标：在 v0.5 已完成轻量管理台的基础上，建立可审计的 Parity Matrix，补齐少量高价值、边界清晰、可验收的功能缺口，完成真实构建、真机/模拟器回归、UI 统一、安全审计和稳定发布准备。

## 2. 已完成版本能力总结

### 2.1 v0.1 已完成能力

v0.1 建立了最小闭环：

1. 多实例管理：添加、删除、切换、测试连接。

## 3. v1.0 版本定位

v1.0 是第一个面向稳定发布的版本。

它不是“再做一个纯功能版本”，也不是“一次性复刻完整 Web 管理台”。v1.0 的定位是：

```text
v1.0 = 有限功能补齐 + 原子级对齐 + 真机回归 + 稳定发布准备
```

本版本必须证明：

1. v0.1~v0.5 的主流程在真实设备或模拟器上可运行。
2. 已知的 P0/P1 行为差异被记录、处理或明确接受。
3. 新增补齐功能不破坏已完成版本能力。
4. App 的权限、安全、缓存、副作用、错误语义与 Web/后端行为有明确对照。
5. 可以产出 v1.0.0 APK，并具备发布说明、已知限制和回归证据。

## 4. v1.0 范围边界

### 4.1 范围制定原则

v1.0 的功能补齐必须同时满足：

1. 已有基础架构或 Repository 能自然扩展。
2. REST API 或后端源码可确认，不靠猜测实现关键行为。
3. 能在执行计划中拆成可验收任务。
4. 能通过单元测试、构建、真机/模拟器和自建 OpenList 实例验证。
5. 不引入新的大模块形态或长期不可维护的特殊路径。

### 4.2 v1.0 必做范围

#### A. 原子级对齐与发布稳定化

1. 创建并维护 `Parity_Matrix.md`。
2. 对 v0.1~v0.5 全部主流程建立回归清单和结果记录。
3. 在本地构建环境执行完整构建与单元测试。
4. 在真机或模拟器上完成安装、启动、登录、浏览、上传、下载、分享、搜索、任务、预览、管理台回归。
5. 验证 Room 8→9 真实升级路径。
6. 验证 v0.4/v0.5 遗留的 Media3、Markdown、图片预览、暗色模式、Web 管理台兜底等真机项。
7. 统一 UI 细节、错误文案、空态、加载态、危险确认和状态 badge。
8. 完成安全审计：Token、分享密码、签名 URL、私密设置、日志脱敏、Web 兜底。
9. 更新 `README.md`、`RELEASE_NOTES.md`、`KNOWN_ISSUES.md`、`VERSION_STATUS.md`。
10. 版本号升级为 `versionCode = 6`、`versionName = "1.0.0"`。

#### B. 登录与权限补齐

1. 增加 LDAP 登录入口和 Repository 接线。
2. 增加 2FA/OTP 最小闭环：当后端返回需要 OTP 时，展示 OTP 输入并完成登录。
3. 登录页明确区分账号密码、LDAP、游客、Token 登录。
4. SSO/WebAuthn 不做原生化，但必须提供安全的浏览器兜底说明或 Web 管理台/Token 手动回退路径。
5. 文件写入口从“仅游客隐藏、已登录乐观展示”升级为“目录级 `write` 字段 + 会话权限位 + 后端 403 兜底”的组合门控。
6. 目录级权限不可确认时，保留乐观展示但在 UI 上保持低风险文案，后端 403 仍为最终裁决。

#### C. 上传、下载与任务补齐

1. 上传失败任务支持一键重试。
2. 本地下载任务支持从任务中心取消。
3. 本地下载任务失败原因展示与刷新逻辑统一到任务中心。
4. 任务中心补充本地上传/下载与远程任务的统一错误文案和空态。
5. 不做自研下载器、断点续传、分片上传、多线程下载。

#### D. 分享与链接补齐

1. 支持识别同实例 OpenList 分享链接，并在 App 内进入受控的分享打开流程。
2. 分享链接入站支持两类来源：
   - Android `ACTION_VIEW` 打开同实例分享 URL。
   - App 内复制/输入分享 URL 后解析。
3. 入站分享最小闭环包括：识别实例、识别分享 id、请求分享信息、展示分享基础信息、复制链接、外部浏览器打开。
4. 对可以映射到普通文件路径且当前会话有权限的分享文件，复用 v0.4 预览/外部打开分发。
5. 分享密码仅用于本次请求，不写入持久化缓存，不进入日志。
6. 不做分享态上传、分享态完整目录写操作、分享访问统计、二维码生成。
7. 若后端/API 行为无法可靠确认，必须降级为外部浏览器打开并在 Parity Matrix 记录差异。

#### E. 预览与播放器补齐

1. Markdown 内嵌图片做最小支持：同实例相对路径图片可加载，失败时显示占位，不阻断正文渲染。
2. Markdown 外部图片不自动带 Token，不注入敏感 Header。
3. 视频、音频、字幕、图片预览完成真机验证，并将不支持格式写入 `KNOWN_ISSUES.md`。
4. 字幕 `max` 支持范围、加载失败文案和关闭字幕状态统一。
5. 不做代码高亮、歌词、图片画廊模式、PDF/Office 内置预览。

#### F. 管理台轻补齐

1. 索引更新支持选择路径：复用 v0.2 目录选择器，默认 `/` 仍保留。
2. 索引 `maxDepth` 行为必须在真实实例或后端源码中验证；若 `-1` 不可靠，执行计划必须提出替代默认值。
3. 管理任务增加后端已存在的批量轻操作入口，范围限于：
   - 清理已完成或已成功记录。
   - 重试失败任务。
   - 批量操作前必须展示影响范围并二次确认。
4. 管理台设置页保留只读，不做保存/删除/重置 API Token。
5. 用户、存储、驱动的创建/编辑/删除仍走 Web 管理台兜底。
6. Web 兜底继续使用外部浏览器，不注入 Token/Cookie。

### 4.3 v1.0 可选范围

## 5. 原子级对齐定义与 Parity Matrix

### 5.1 v1.0 对齐定义

v1.0 的“原子级对齐”不等于 UI 像素级复刻，而是要求以下维度可追踪、可验收：

1. 功能入口：Web 有的核心操作，Android 是否有等价入口或明确兜底。
2. API 契约：接口、方法、请求体、Header、响应字段是否一致。
3. 权限语义：游客、普通用户、管理员的可见入口和后端裁决是否一致。
4. 状态语义：上传、下载、分享、任务、索引、预览状态是否一致。
5. 错误语义：401、403、404、签名失效、搜索不可用、管理拒绝等是否一致。
6. 副作用：写操作、存储启停、任务操作、索引操作后的缓存失效和刷新范围是否一致。
7. 安全：Token、Cookie、签名 URL、私密设置、分享密码是否不泄漏。
8. 差异记录：无法原生实现或本版本不做的能力必须有 Web 兜底或明确 Deferred 说明。

### 5.2 `Parity_Matrix.md` 要求

v1.0 必须创建 `Parity_Matrix.md`。

每一行至少包含：

| 字段 | 说明 |
| --- | --- |
| Area | Auth / Instance / Files / Upload / Download / Share / Search / Task / Preview / Admin / Settings |
| Web 操作 | Web 端对应行为 |
| Android 入口 | App 内页面、按钮或菜单 |
| API / 路径 | REST API、下载路径或 Web fallback 路径 |
| 权限 | Guest / User / Admin |
| 成功结果 | 成功后的状态、刷新、副作用 |
| 失败结果 | 错误码、错误 message、UI 展示 |
| 缓存影响 | 失效范围或不缓存 |
| Android 状态 | Done / Partial / Web Fallback / Deferred / Not Applicable |
| 差异说明 | 具体差异与原因 |
| v1.0 处理 | 修复 / 接受 / 兜底 / 延后 |
| 验收证据 | 单测、构建、真机步骤、截图/日志引用 |

### 5.3 Parity 覆盖范围

v1.0 必须覆盖：

1. 实例管理。
2. 登录：账号密码、LDAP、2FA、游客、Token。
3. 文件列表、详情、路径、排序/刷新。
4. 上传与上传任务。
5. 下载与下载任务。
6. 新建、重命名、删除、移动、复制、批量操作。
7. 分享创建、列表、详情、编辑、启停、删除、入站链接。
8. 搜索和搜索不可用状态。
9. 任务中心：本地上传、本地下载、远程任务、离线下载。
10. 预览：图片、文本、Markdown、视频、音频、字幕、PDF/Office/未知格式兜底。
11. 管理台：门控、用户、存储、任务、索引、设置、Web 兜底。
12. 缓存、安全、错误、权限、副作用。

## 6. UI/UX 要求

## 7. API 需求

### 7.1 API 接入原则

v1.0 继续遵守：

1. UI 层不得直接调用 Retrofit。
2. 所有 API 必须通过 Repository。
3. Repository 返回领域模型，不向 UI 暴露 DTO。
4. 所有实例数据必须带 `instanceId`。
5. Token 由 `TokenProvider` / Interceptor 处理，禁止 UI 拼 Header。
6. 路径统一经 `OpenListPathCodec`。
7. 写操作默认不自动重试，除非明确幂等。
8. 后端 message 不得被无意义吞掉；操作类失败优先保留原始 message。

### 7.2 v1.0 需要新增或强化的接口

#### 登录与会话

1. `/api/auth/login/ldap`：LDAP 登录。
2. 2FA/OTP 相关接口：以执行计划 S0 从后端源码或官方 API 文档核对为准，不允许凭空猜字段。
3. `/api/me`：登录后、管理台门控和权限刷新继续复用。

#### 文件权限

1. `/api/fs/list`：读取目录级 `write` 或等价权限字段。
2. `/api/me`：读取用户权限位，补充入口展示判断。
3. 文件操作接口继续以后端 403 为最终裁决。

#### 上传、下载、任务

1. 上传重试复用已有上传接口和 `UploadTaskEntity`。
2. 本地下载取消复用系统 `DownloadManager.remove`。
3. 下载状态刷新复用现有 DownloadManager 查询。
4. 任务中心不新增后端任务类型，除管理台批量任务接口外不扩大远程任务拉取范围。

#### 分享入站

执行计划必须核对以下行为后定稿：

1. 分享 Web URL 格式，如 `/@s/{sid}`。
2. 分享下载路径，如 `/sd/:sid/*path`。
3. 分享详情 API 是否支持匿名/密码访问，字段和错误语义如何。
4. 若现有 API 不足以原生安全打开，必须降级 Web fallback。

#### 预览

1. Markdown 内嵌同实例图片需要获取安全可加载 URL。
2. 不对外部图片注入 Authorization。
3. 媒体 URL 继续采用进入页面时刷新 `fs/get` 的策略。

#### 管理台

1. `admin/index/update`：补齐 `paths` 选择和 `maxDepth` 校准。
2. `admin/task/*/cancel_some`、`delete_some`、`retry_some`、`clear_done`、`clear_succeeded`、`retry_failed`：执行计划必须先核对后端接口存在性、权限、请求体和副作用，再决定落地的 UI 子集。
3. `@manage` Web 兜底继续只构造同源路径。

如执行计划需要自动化辅助，可新增测试工具、脚本或文档模板，但不得为了 Parity Matrix 在 App 内引入无关页面或持久表。

## 9. 数据模型需求

### 9.1 数据库原则

v1.0 优先复用既有表，只有在必须持久化新状态时才新增字段或表。

新增 Migration 必须：

1. 明确从当前 Room 版本继续递增。
2. 提供手写 SQL。
3. 导出 schema JSON。
4. 用 KSP 重新生成并核对 identityHash。
5. 在真机/模拟器上验证 v0.5.0 数据升级到 v1.0.0 后不丢失。

### 9.2 可能涉及的数据模型

#### Auth

1. `AuthMethod`：PASSWORD / LDAP / TOKEN / GUEST。
2. `OtpChallenge`：只在内存态保存，不持久化。
3. `LoginResult`：支持 SUCCESS / NEED_OTP / FAILURE。

#### Permission

1. `DirectoryCapability`：目录级 `canWrite`、`canUpload`、`canShare` 等展示能力，字段以真实 API 为准。
2. 不把目录权限作为安全边界，后端 403 仍为最终裁决。

#### Upload / Download

1. `UploadTaskEntity` 可复用，必要时补充 retry metadata。
2. `DownloadTaskEntity` 可复用，必要时补充 cancel/error 状态，但优先使用既有字段。
3. 不新增断点续传字段，避免虚假承诺。

#### Share Inbound

1. `ShareInboundTarget`：instanceId、baseUrl、sid、path?、requiresPassword?、sourceUrl。
2. 分享密码不持久化。
3. 分享入站结果可走内存态，不强制建表。

#### Preview

1. Markdown 图片不直接落库保存原始 signed URL。
2. 如需缓存图片，必须沿用 Coil/preview 缓存隔离策略，cache key 带 `instanceId`。

#### Admin

1. `AdminIndexUpdateRequest`：paths、maxDepth。
2. `AdminTaskBatchResult`：operation、requestedCount、successCount、failedCount、failedItems。
3. 不扩展 `RemoteTaskEntity` 来保存管理任务批量结果。

## 10. 缓存与刷新要求

### 10.1 文件和目录缓存

1. 文件写操作成功后继续精确失效受影响目录。
2. 上传重试成功后刷新目标目录。
3. 分享入站不应污染当前实例普通目录缓存，除非确认其映射到普通路径且当前会话具备权限。
4. 目录级权限刷新应随 `fs/list` 一起更新，不另建长期缓存。

### 10.2 上传下载任务缓存

1. 上传重试后更新原任务状态，不重复创建无关联任务。
2. 下载取消后状态必须更新为 CANCELLED 或等价状态。
3. DownloadManager 状态与本地 Room 记录不一致时，以系统状态刷新结果为准。

### 10.3 预览缓存

1. Markdown 内嵌图片失败不导致 Markdown 正文缓存失效。
2. 文件重命名、删除、移动、复制、上传覆盖仍需要联动失效预览缓存。
3. 存储启停仍按 v0.5 精确/保守策略失效文件和预览缓存。

### 10.4 管理缓存

1. `admin_cache` 继续只缓存用户、存储、设置。
2. 管理任务批量操作不入库，操作成功后刷新内存态。
3. 索引路径更新成功后刷新索引进度，不假设立刻完成。
4. Web 兜底不缓存任何 Web 会话信息。

## 11. 权限与安全要求

### 11.1 Android 权限

v1.0 不应新增高风险权限。

继续使用：

1. `INTERNET`。
2. `POST_NOTIFICATIONS`。
3. 既有上传/下载所需的 SAF、WorkManager、前台/后台能力。

不得新增：

1. `MANAGE_EXTERNAL_STORAGE`。
2. Camera。
3. Contacts。
4. Location。
5. SMS/Phone。

如执行计划提出任何新增权限，必须实时决策并说明原因、影响和验收方式。

### 11.2 Token 与敏感信息

1. Token 继续使用 Keystore 加密保存。
2. OTP 只在内存态使用，不持久化，不写日志。
3. 分享密码只在本次请求或 UI 临时态使用，默认不持久化。
4. 签名 URL、Authorization、Cookie、Token、password、secret、key、sign 参数必须脱敏。
5. 私密设置值继续在领域模型构造前置空，不进入 `admin_cache.rawJson`。
6. Web 兜底不注入原生 Token/Cookie。

### 11.3 管理操作安全

1. 存储启停、重新加载全部、索引清空、批量任务操作必须二次确认。
2. 批量任务操作必须展示影响范围和不可逆说明。
3. 非管理员进入管理台仍必须零 admin 请求。
4. 管理 API 拒绝访问必须映射为明确的权限错误，不误导为网络错误。

### 11.4 分享入站安全

1. 仅自动处理已配置实例同源的分享链接。
2. 未配置实例、跨域实例或无法解析的链接，必须提示用户添加实例或外部浏览器打开。
3. 不允许从任意分享 URL 中提取并保存 Token。
4. 不允许对外部图片或外部链接注入 Authorization。

## 12. 错误处理要求

### 12.1 新增或强化错误类型

执行计划可按需要新增或复用以下错误：

1. `AuthMethodUnavailable`：LDAP/OTP/SSO 等登录方式当前实例不可用。
2. `OtpRequired`：登录需要 OTP。
3. `OtpInvalid`：OTP 错误或过期。
4. `DirectoryPermissionUnknown`：目录权限字段不可确认。
5. `UploadRetryUnavailable`：上传任务无法重试。
6. `DownloadCancelUnavailable`：系统下载任务无法取消。
7. `ShareLinkUnsupported`：分享链接无法原生解析。
8. `SharePasswordRequired`：分享需要密码。
9. `AdminBatchOperationFailed`：管理批量操作部分或全部失败。

### 12.2 错误展示规则

1. 后端 message 可信且具体时优先展示。
2. 本地错误必须提供可操作下一步，如重试、刷新、外部浏览器打开、复制链接、返回。
3. 权限错误不能伪装成空态。
4. 搜索未建索引、分享密码缺失、管理拒绝访问必须有专属说明。
5. 批量操作部分失败必须提供失败项列表。
6. 预览失败不得导致 App 崩溃，必须有下载/外部打开/Web 打开兜底。

### 13.1 构建与测试

## 16. 后续版本预留

v1.0 后续可继续评估：

1. 完整用户/存储 CRUD。
2. 动态驱动配置表单。
3. 完整设置编辑和 API Token 原生重置。
4. Meta、消息、扫描管理。
5. 分享二维码、访问统计和审计。
6. 完整分享态目录浏览和上传。
7. PDF/Office 内置预览。
8. 代码高亮、歌词、图片画廊。
9. 归档浏览、Torrent 完整能力。
10. 自研下载器、断点续传、分片上传。
11. SSO/WebAuthn 原生实现。
12. Play 商店发布。

这些能力不进入 v1.0 强验收，最多在 `Parity_Matrix.md` 中标记为 Deferred 或 Web Fallback。

## 17. 成功标准

v1.0 成功的标准是：

1. 不是功能数量最多，而是稳定、可信、可回归。
2. v0.1~v0.5 主流程在真实运行环境中通过。
3. B 方案的有限功能补齐完成且不破坏已完成能力。
4. Parity Matrix 清楚说明哪些已对齐、哪些接受差异、哪些走 Web 兜底、哪些延期。
5. 所有高风险已知 caveat 都有验证结果或明确保留说明。
6. UI、错误、安全、缓存、副作用的行为可解释、可复现。
7. 可以产出 v1.0.0 APK、发布说明和验收报告。

---

## 18. 推荐结论

推荐确认 v1.0 范围为：

```text
v1.0 = Parity Matrix + 真机回归 + UI/安全/缓存/错误统一
     + 登录补齐（LDAP/2FA）
     + 权限门控补齐
     + 上传重试 / 下载取消
     + 分享链接入站最小闭环
     + Markdown 内嵌图片最小支持
     + 管理台索引路径选择 / 批量任务轻操作
     + v1.0.0 稳定发布准备
```

不推荐在 v1.0 同时加入完整 Web 管理台 CRUD、动态驱动表单、完整设置编辑、完整分享态目录写操作、完整离线同步或自研下载器。这些能力会把 v1.0 从稳定发布版拖回长期功能扩张版，导致 Parity Matrix 和真机回归无法收口。

本 PRD 已按用户选择的 B 方案制定：有补齐，但补齐必须有限、有证据、可验收。
