### 5.2 Room 数据库迁移

Status: **Accepted（手写 SQL + 逐字段核对通过；schema 结构已用 KSP 实际核对并修复真实 identityHash；真机升级路径仍未验证）**

Evidence:
- `OpenListDatabase` 升级到 version 9，新增 `AdminCacheEntity`（`id/instanceId/scope/cacheKey/rawJson/cachedAt` + `(instanceId,scope,cacheKey)` 唯一索引）。
- `MIGRATION_8_9` 手写 SQL 与手工编写的 `schemas/.../9.json` 逐字段核对一致。
- **本轮环境可运行 Gradle**，S8 阶段执行了 `:core:database:kspDebugKotlin` 强制重新生成 `9.json`：**与手工推导版本逐字段比对完全一致，仅 `identityHash` 从占位符变为真实值**（`edad8842c1f0268bcb41ace48375b2d7`）——证明手工推导的表结构/索引/列定义本身完全正确。**已将该真实 `identityHash` 写回并提交至仓库的 `9.json`**（不再是占位符），比 v0.4 报告中"未经 KSP 验证"更进一步彻底解决。剩余 gap 收窄为"真机上执行迁移、旧数据保留"这一项，非 schema 描述本身的正确性。
- `8.json` 的 `identityHash` 占位符（`PLACEHOLDER_UNVERIFIED_8_00000000000000`，v0.4 遗留）**本轮判定为无法修复**：KSP 只导出当前数据库版本（9）的 schema，要核对历史版本 8 需临时把 `OpenListDatabase.version` 降回 8 重新编译，侵入性/风险与收益不匹配，故保留原占位符，不阻塞发布（详见 `KNOWN_ISSUES.md`）。
- `InstanceRepositoryImpl.delete` 级联清理扩展到 `admin_cache` 表（`adminCacheDao.deleteByInstanceId`）。

## 7. S8 Stabilization: Two Independent Audits + Regression Review

### 7.1 审计一：缓存/多实例隔离/敏感字段处理

**结论：发现问题数 0**

- 全部 7 个 admin Repository 的每一次 `admin_cache` 查询都按 instanceId 限定：复合键（`instanceId+scope+cacheKey`）+ WHERE 子句 + 唯一索引三重保证，跨实例数据碰撞在结构上不可能发生。
- `InstanceRepositoryImpl.delete()` 调用 `adminCacheDao.deleteByInstanceId`，删除实例即清理其全部管理缓存。
- 全部 7 个 Repository 的每次调用都做新鲜的 `instanceRepository.getById(instanceId)` + `instanceContext.set(...)`，不存在跨调用持有的过期 instance 状态。
- `AdminTaskRepositoryImpl` 的内存态 StateFlow 键为 `"$instanceId:$taskType:$bucket"`，`observeAdminTasks` 按 instanceId 前缀过滤——确认切换实例导航后无状态残留。
- 全部 7 个 Tab ViewModel 均采用幂等 `bind(instanceId)` 模式，经 `LaunchedEffect(instanceId)` 触发重置，确认切实例不跨实例污染状态。
- 敏感字段处理：`AdminUserRepositoryImpl`/`AdminStorageRepositoryImpl`/`AdminSettingsRepositoryImpl` 全部对**映射后的领域模型**调用 `json.encodeToString(domain)`，从不对 DTO 调用（审计确认代码库中不存在任何 `json.encodeToString<AdminXxxDto>` 调用点）；私密设置值在 `AdminSettingDto.toDomain()` 内、对象构造**之前**即被置空（不是"先缓存再脱敏"）；单元测试直接断言缓存的 `rawJson` 字符串不含测试用密钥子串；`core/model/AdminModels.kt` 全部字段经审查确认无 password/hash/salt/secret 类字段。

### 7.2 审计二：权限门控/日志脱敏/回归面

**结论：发现问题数 0**

- `AdminHostScreen.kt` 的 `when(accessState)` 是穷尽分支，7 个 Tab 组件只在 `ALLOWED` 分支内被引用——结构性不可达（编译期分支保证），非仅运行时隐藏。
- `AdminViewModel.init{}` 无条件调用 `checkAccess()`，与导航路径无关（深链/返回/恢复均命中同一道闸门）。
- 全部 7 个 Tab ViewModel 均为空构造函数 + 显式 `bind()`，`init{}` 内不发起任何网络调用（门控完成前零请求）。
- 401 中途失效触发 `sessionManager.invalidate` + 响应式 `observeAccess()` 转为 `SESSION_EXPIRED`；403 从任何管理操作触发均经 `DomainError.OpenListError`/`AdminAccessDenied` 携带后端原始 message 展示，从未被静默吞掉或误报为成功。
- 日志脱敏：v0.5 admin 代码全程零 `Log.`/`Timber`/日志调用；项目全局本就没有配置 `HttpLoggingInterceptor`（S8 确认这是项目级预先存在的状态，非 v0.5 引入的新缺口，已记入 `KNOWN_ISSUES.md` 作为一般性观察）；Web 兜底的 `ACTION_VIEW` Intent URL 从不被记录且从不含 Token（`AdminWebFallbackRepositoryImpl` 构造逻辑确认）。
- 回归面：`git diff --stat` 对 `TaskRepositoryImpl.kt`/`TaskAggregationRepositoryImpl.kt`/`feature/task/`/`feature/files/`/`feature/preview/`/`feature/search/`/`feature/share/`/`feature/upload/` 全部为零改动（本报告 §7.3 附完整 15 项回归清单核实过程）；`AndroidManifest.xml` 仍是 4 个权限（`INTERNET`/`POST_NOTIFICATIONS`/`FOREGROUND_SERVICE`/`FOREGROUND_SERVICE_DATA_SYNC`），零新增，不含任何禁止清单里的危险权限（`MANAGE_EXTERNAL_STORAGE`/Camera/Contacts/Location/SMS/Phone）。
