# OpenList Android 原生客户端 v0.4 需求文档

版本：v0.4
阶段定位：预览与播放器闭环版
前置版本：v0.1、v0.2、v0.3 已完成开发、执行与验收
目标平台：Android
推荐技术栈延续：Kotlin + Jetpack Compose + Retrofit/OkHttp + Room + DataStore + WorkManager
UI/UX 约束：继续遵守根目录 `DESIGN.md`
核心目标：在 v0.3 已完成的“分享 + 搜索 + 统一任务中心 + 基础离线下载”基础上，补齐常见文件预览、媒体播放和外部打开能力，让用户在移动端完成“找到文件 → 打开/预览/播放 → 下载或外部处理”的闭环。

### 1.3 长期产品目标

OpenList Android 原生客户端的长期目标是实现真正的 Android 原生 APK，而不是 WebView 包壳。

长期目标包括：

1. 多实例管理。
2. 登录、游客访问、管理员访问。
3. 文件浏览与文件详情。
4. 上传、下载和文件写操作。
5. 搜索、分享、任务中心和离线下载。
6. 图片、文本、Markdown、PDF、Office 等预览。
7. 视频和音频播放。
8. 字幕、歌词、代码高亮、图片画廊等增强预览能力。
9. 轻量管理台。
10. 与 Web 端功能、权限、错误语义、状态、副作用进行原子级对齐。

### 1.4 已确认版本节奏

已确认渐进路线：

```text
v0.1：技术验证与最小闭环
v0.2：上传与文件操作
v0.3：分享、搜索与统一任务中心
v0.4：预览与播放器
v0.5：轻量管理台
v1.0：原子级对齐稳定版
```

v0.4 只能实现“预览与播放器”范围内的能力；管理台、归档、Torrent、完整分享态浏览等长期能力只能做架构预留，不进入本版本强验收。

## 3. v0.4 版本定位

v0.4 是“预览与播放器闭环版”。

一句话定义：

```text
v0.4 = 文件打开分发 + 图片预览 + 文本预览 + Markdown 预览 + PDF/Office 外部打开 + 视频播放器 + 音频播放器 + 基础字幕支持
```

v0.4 的用户价值是：

1. 用户在文件列表、文件详情、搜索结果、分享详情和任务完成结果中，可以直接打开可预览文件。
2. 常见图片可以在 App 内原生查看。
3. 常见文本和 Markdown 文件可以在 App 内阅读。
4. 视频和音频可以在 App 内播放。
5. 不适合内置渲染的 PDF、Office 和未知格式可以稳定交给外部应用或 Web 兜底。
6. v0.5 管理台和 v1.0 对齐阶段可以复用统一的文件打开分发和媒体能力。

## 4. v0.4 需要决策的事项

本章节列出当前推荐决策。执行计划或实现阶段如发现新的关键决策，必须实时向用户汇报。

### D-01：v0.4 是否将 `feature/preview` 从占位升级为真实模块？

推荐决策：是。

理由：

1. v0.1 和 v0.2 已为预览能力预留 `feature-preview` / `feature/preview`。
2. v0.4 是预览与播放器版本，继续留在 `:app` 占位包会破坏多模块连续性。
3. 文件打开分发、图片/文本/Markdown/媒体 UI 都应有清晰模块边界。

影响：

1. 新增 `:feature:preview`。
2. 删除或迁移 `:app` 内 preview 占位包。
3. v0.4 执行计划需定义模块依赖方向。

### D-02：PDF 是否内置预览？

推荐决策：否。v0.4 只做 PDF 外部打开。

理由：

1. `ROADMAP.md` 明确 v0.4 是“PDF / Office 外部打开”。
2. `Full_PRD.md` 允许 PDF 外部或内置预览，但长期方向不能扩大当前版本范围。
3. 内置 PDF 预览会引入额外依赖、分页、缩放、权限和内存问题。

影响：

1. PDF 文件显示“外部打开”和“下载”。
2. 后续版本可评估内置 PDF 预览，但不计入 v0.4 验收。

### D-03：Office 是否内置预览？

推荐决策：否。v0.4 只做 Office 外部打开。

理由：

1. `ROADMAP.md` 明确 Office 为外部打开。
2. Office 内置预览依赖复杂，且不同格式兼容性差。
3. 移动端更合理的 v0.4 闭环是识别格式并交给系统或 Web 兜底。

影响：

1. Word、Excel、PowerPoint 等文件不进入 App 内渲染。
2. 无外部应用可处理时，提示下载或 Web 打开。

### D-04：视频/音频播放采用内置播放器还是系统 Intent？

推荐决策：内置播放器为主，系统外部播放器为兜底。

理由：

1. v0.4 路线明确包含视频播放器和音频播放器。
2. `Full_PRD.md` 对媒体播放的长期方向是原生播放器体验。
3. 各存储驱动直链、代理、Range 支持存在差异，必须保留外部打开兜底。

影响：

1. 需要统一媒体播放封装。
2. 播放失败或格式不支持时，必须可下载、外部打开或 Web 打开。

### D-05：字幕支持做到什么程度？

推荐决策：v0.4 做基础外挂字幕支持，不做字幕管理器。

范围：

1. 支持同目录同名字幕的自动发现，若接口和列表缓存可支撑。
2. 支持用户从当前目录选择字幕文件。
3. 支持常见字幕格式的最小接入。
4. 不做字幕搜索、在线下载、字幕样式编辑、手动时间轴校准。

影响：

1. 需要在数据模型中表达字幕候选。
2. 需要在播放器 UI 中提供字幕开关和选择入口。

### D-06：v0.3 延期的“App 内打开分享 URL”是否纳入 v0.4 必做？

推荐决策：不纳入 v0.4 必做，仅复用分享详情中的已知文件预览入口。

理由：

1. `ROADMAP.md` 的 v0.4 范围是预览与播放器，不是分享态浏览。
2. 完整解析和打开任意分享 URL 会涉及分享态鉴权、密码、目录浏览和上传权限，容易扩大范围。
3. v0.3 已有分享详情和文件路径信息，v0.4 可以先支持从已知文件入口预览。

影响：

1. v0.4 不做完整分享态目录浏览。
2. 任意分享 URL 入站解析继续作为后续预留。

## 6. UI/UX 设计要求

### 6.1 与 `DESIGN.md` 的关系

v0.4 所有新增页面、弹窗、底部 Sheet、播放器控制、错误态、空态和操作按钮必须遵守 `DESIGN.md`。

本 PRD 不复制 `DESIGN.md` 的颜色、字体、圆角、间距、组件细节。执行计划和实现阶段应直接引用 `DESIGN.md` 与现有 `core:designsystem` 组件。

### 6.2 设计系统使用原则

1. v0.4 不重新定义设计系统。
2. v0.4 新增组件优先复用现有 `core:designsystem`。
3. 必须新增组件时，应在执行计划中说明与 `DESIGN.md` 的映射关系。
4. 预览页和播放器页需要适配浅色、深色和沉浸式场景。
5. 危险、失败、加载、空态、无权限等状态必须复用现有语义体系。
6. 移动端触控目标、按钮密度和列表可读性必须按 `DESIGN.md` 验收。

### 6.3 v0.4 新增或完善的通用组件

v0.4 至少需要以下组件或能力：

1. `FileOpenDispatcher`：根据文件类型、MIME、扩展名和后端能力决定打开方式。
2. `PreviewScaffold`：统一预览页外壳，承载标题、路径、操作入口和错误态。
3. `ImagePreviewSurface`：图片预览区域。
4. `TextPreviewSurface`：文本预览区域。
5. `MarkdownPreviewSurface`：Markdown 预览区域。
6. `MediaPlayerSurface`：视频/音频播放器区域。
7. `SubtitleSelector`：字幕选择入口。
8. `ExternalOpenSheet`：外部打开、下载、Web 打开兜底操作。
9. `PreviewErrorState`：预览失败、格式不支持、无权限、签名失效等状态。
10. `PreviewLoadingState`：预览加载中状态。

### 6.4 新增页面和入口的 UI 要求

1. 文件列表项或操作菜单必须提供“预览/打开”入口。
2. 文件详情页必须提供清晰的主操作，优先打开可预览文件。
3. 搜索结果中的文件点击可进入预览或详情，具体交互需在执行计划中定稿。
4. 分享详情中的文件可通过统一分发入口预览。
5. 任务中心中已完成且指向文件的任务可打开文件。
6. 所有预览页必须提供下载或外部打开兜底。
7. 媒体播放页必须支持返回、播放/暂停、进度拖动、错误提示。

## 7. 用户场景

### 7.1 图片预览

用户在文件列表、文件详情或搜索结果中点击图片文件，App 直接打开图片预览页。

验收要求：

1. 支持常见图片格式。
2. 加载中有明确状态。
3. 加载失败时可重试、下载或外部打开。
4. Token 失效时按既有会话策略处理。
5. 图片预览不影响原有浏览和下载链路。

### 7.2 文本预览

用户点击文本文件，App 打开文本预览页。

验收要求：

1. 支持常见纯文本文件。
2. 大文件必须避免一次性加载导致卡顿或崩溃。
3. 文本内容只读。
4. 支持复制文本或选择复制作为可选能力，不强制。
5. 不支持编码或读取失败时必须提供兜底操作。

### 7.3 Markdown 预览

用户点击 Markdown 文件，App 打开 Markdown 预览页。

验收要求：

1. Markdown 内容可渲染为阅读视图。
2. Markdown 渲染不得执行危险脚本。
3. 远程图片、链接等内容需要有安全和失败兜底。
4. 可切换源码视图作为可选能力，不强制。
5. Markdown 缓存策略必须独立于目录缓存。

### 7.4 PDF / Office 外部打开

用户点击 PDF 或 Office 文件时，App 不内置渲染，而是提供外部打开流程。

验收要求：

1. 系统存在可处理 App 时，可以跳转外部应用。
2. 无可处理 App 时，提示下载或 Web 打开。
3. 外部打开所使用的 URI 不泄露 Token。
4. 短期签名 URL 失效时可以刷新或提示用户重试。
5. 外部打开失败不导致 App 崩溃。

### 7.5 视频播放

用户点击视频文件，App 打开内置视频播放器。

验收要求：

1. 支持播放、暂停、进度拖动。
2. 支持横屏播放。
3. 播放失败时可外部打开或下载。
4. 网络中断、403、404、签名失效等错误有明确提示。
5. 不支持转码，不保证所有驱动和格式都能内置播放。

### 7.6 音频播放

用户点击音频文件，App 打开音频播放器。

验收要求：

1. 支持播放、暂停、进度拖动。
2. 支持显示文件名、路径和基础状态。
3. 播放失败时可外部打开或下载。
4. 不做歌词显示。
5. 后台播放通知控制可作为可选增强，不作为 v0.4 强验收。

### 7.7 字幕支持

用户播放视频时，可选择可用字幕。

验收要求：

1. 支持同目录字幕候选发现或手动选择同目录字幕。
2. 字幕文件加载失败时不影响视频主播放。
3. 字幕入口必须允许关闭字幕。
4. 不做字幕下载、搜索、样式编辑和时间轴校准。

## 8. 权限与状态规则

### 8.1 游客态

游客态允许预览后端允许访问的公开文件。

规则：

1. 游客态不显示需要登录的写操作。
2. 游客态打开预览时不附带 Token。
3. 后端返回 401/403 时显示登录引导或无权限提示。
4. 游客态不得访问非公开分享或私有路径。

### 8.2 普通登录用户

普通登录用户可以预览其有权限读取的文件。

规则：

1. 读取权限以后端返回为准。
2. 客户端可以乐观展示预览入口，但必须正确处理 403。
3. 预览失败不应影响文件列表缓存。
4. 下载、外部打开和 Web 打开必须尊重后端权限。

### 8.3 管理员 Token 用户

管理员 Token 用户可预览其 Token 可访问的文件。

规则：

1. 不因管理员身份跳过后端权限检查。
2. 不在日志、错误文案或外部 Intent 中泄露 Token。
3. 多实例之间 Token 和预览缓存必须隔离。

### 8.4 Token 失效

规则：

1. 预览请求返回 401 时，复用既有会话失效处理。
2. 媒体播放过程中签名 URL 失效时，优先尝试刷新可播放地址。
3. 刷新失败后展示错误并提供返回、下载或外部打开兜底。
4. 不应因为播放器请求失败导致全局崩溃。

## 9. API 需求

### 9.1 API 接入原则

v0.4 必须延续既有 API 接入原则：

1. UI 层不得直接调用 Retrofit。
2. 所有 API 必须通过 Repository。
3. Repository 返回领域模型，不向 UI 暴露 DTO。
4. 所有错误统一映射为 `DomainError` 或其扩展类型。
5. 所有实例相关请求必须携带 `instanceId`。
6. 路径编码必须继续走统一路径工具。
7. 不把签名 URL 当作长期稳定数据缓存。

### 9.2 v0.4 需要正式启用或验证的接口/路径

v0.4 需要围绕以下后端能力建立稳定封装：

1. 文件详情接口，用于获取文件元信息、类型、大小、修改时间和可能的直链信息。
2. 文件列表接口，用于同目录字幕候选发现。
3. 下载或代理路径，用于图片、文本、Markdown、视频、音频内容读取。
4. 分享相关路径，用于分享详情中的已知文件预览入口。
5. 任务中心已完成任务中的路径跳转能力。

具体接口路径、请求方法、响应字段和下载/代理路径选择必须在 `v0.4_EXECUTION_PLAN.md` 中通过后端源码、官方 API 文档或真实实例行为进一步确认。

### 9.3 内容读取策略

不同文件类型应使用不同读取策略：

1. 图片：优先使用可鉴权或签名的图片 URL。
2. 文本：必须支持可取消请求；大文件应采用流式、分页或大小限制策略。
3. Markdown：读取原始 Markdown 文本后在本地安全渲染。
4. 视频/音频：优先使用可被播放器消费的 URL；需要处理 Range、302、403、签名失效和驱动不支持。
5. PDF/Office：生成可外部打开的临时 URI 或外部 Intent，不做本地解析。

### 9.4 接口行为验证要求

执行计划中必须验证：

1. `/api/fs/get` 或等价文件详情返回的直链字段及有效期。
2. `/d/*path` 与 `/p/*path` 在登录态、游客态、子路径部署下的行为。
3. 视频/音频 URL 是否支持 Range 请求。
4. 不同驱动对图片、文本、媒体直链的返回差异。
5. 分享详情中已知路径的预览方式。
6. 字幕文件候选的目录列表字段是否足够可靠。
7. 中文、空格、特殊字符路径在预览、播放和外部打开中的编码一致性。

## 10. Repository 设计

### 10.1 新增或正式实现的 Repository

v0.4 至少需要以下 Repository 或领域服务：

1. `PreviewRepository`
2. `MediaRepository`
3. `ExternalOpenRepository` 或等价 helper/service
4. `SubtitleRepository`
5. `PreviewCacheRepository` 或复用 DAO 封装

具体命名可在执行计划中按现有模块风格调整，但必须保持职责清晰。

### 10.2 PreviewRepository

职责：

1. 根据 `instanceId` 和文件路径获取预览描述。
2. 判断文件是否可图片预览、文本预览、Markdown 预览、媒体播放或外部打开。
3. 提供图片、文本、Markdown 的内容来源。
4. 处理文件不存在、无权限、格式不支持、内容过大、签名失效等错误。
5. 返回领域模型，不暴露网络 DTO。

建议方法：

```kotlin
interface PreviewRepository {
    suspend fun resolvePreview(instanceId: String, path: String): ApiResult<PreviewTarget>
    suspend fun loadText(instanceId: String, path: String, options: TextPreviewOptions): ApiResult<TextPreviewContent>
    suspend fun loadMarkdown(instanceId: String, path: String): ApiResult<MarkdownPreviewContent>
    suspend fun refreshPreviewUrl(instanceId: String, path: String): ApiResult<PreviewUrl>
}
```

### 10.3 MediaRepository

职责：

1. 为视频和音频提供可播放媒体源。
2. 处理签名 URL 刷新。
3. 标记驱动或格式不支持的错误。
4. 暴露播放器所需的标题、路径、时长、MIME、字幕候选等领域数据。

建议方法：

```kotlin
interface MediaRepository {
    suspend fun resolveMedia(instanceId: String, path: String): ApiResult<MediaSource>
    suspend fun refreshMediaSource(instanceId: String, path: String): ApiResult<MediaSource>
}
```

### 10.4 SubtitleRepository

职责：

1. 根据视频路径查找同目录字幕候选。
2. 根据用户选择加载字幕。
3. 处理字幕文件格式不支持、无权限、读取失败等错误。

建议方法：

```kotlin
interface SubtitleRepository {
    suspend fun findCandidates(instanceId: String, videoPath: String): ApiResult<List<SubtitleCandidate>>
    suspend fun resolveSubtitle(instanceId: String, subtitlePath: String): ApiResult<SubtitleSource>
}
```

### 10.5 ExternalOpenRepository

职责：

1. 为 PDF、Office、未知格式提供外部打开意图或临时 URI。
2. 对外部打开失败提供可恢复结果。
3. 避免在外部 Intent 中泄露 Token。
4. 提供下载和 Web 打开兜底信息。

建议方法：

```kotlin
interface ExternalOpenRepository {
    suspend fun resolveExternalOpen(instanceId: String, path: String): ApiResult<ExternalOpenTarget>
}
```

### 10.6 与既有 Repository 的关系

1. `FilesRepository` 继续负责文件列表和详情，不直接承担预览渲染职责。
2. `SearchRepository` 的结果点击应交给统一文件打开分发。
3. `ShareRepository` 的分享详情入口应交给统一文件打开分发。
4. `TaskAggregationRepository` 的完成任务点击应交给统一文件打开分发。
5. `TransferRepository` 继续负责下载，不变成预览入口。

## 13. 缓存与刷新要求

### 13.1 预览缓存

需求：

1. 图片缩略图或图片内容缓存可复用图片加载库策略。
2. 文本和 Markdown 可建立轻量缓存。
3. 媒体播放 URL 不长期缓存。
4. 签名 URL 不长期缓存。
5. 预览缓存必须按 `instanceId + path` 隔离。
6. 文件修改后必须刷新预览缓存。

### 13.2 缓存 TTL

建议：

1. 文本/Markdown 缓存 TTL 默认不超过 1 天。
2. 图片缓存可遵循图片加载库默认磁盘缓存策略。
3. 预览元数据缓存可与文件详情缓存保持一致。
4. 用户下拉刷新或文件详情刷新应强制绕过缓存。

### 13.3 与文件操作的缓存联动

需求：

1. 重命名后旧路径预览缓存失效。
2. 删除后对应预览缓存失效。
3. 移动后旧路径缓存失效，新路径按需重新加载。
4. 上传覆盖或同名变化后目标路径预览缓存失效。
5. 删除实例时清理该实例全部预览缓存。

### 13.4 媒体源刷新

需求：

1. 播放源失效时可尝试刷新。
2. 刷新失败时展示错误并提供兜底。
3. 刷新次数应有限制，避免无限重试。
4. 不对写操作自动重试。

## 14. 错误处理要求

### 14.1 新增错误类型

v0.4 需要在错误模型中覆盖：

1. `PreviewUnsupported`：格式不支持内置预览。
2. `PreviewTooLarge`：文本或 Markdown 超过安全读取限制。
3. `PreviewLoadFailed`：预览内容读取失败。
4. `MediaUnsupported`：媒体格式或驱动不支持播放。
5. `MediaSourceExpired`：媒体源或签名 URL 失效。
6. `SubtitleLoadFailed`：字幕加载失败。
7. `ExternalOpenUnavailable`：无外部应用可处理。
8. `ExternalOpenFailed`：外部打开失败。

具体类型命名可在执行计划中按现有 `DomainError` 风格调整。

### 14.2 错误展示规则

1. 无权限：显示权限提示，可引导登录或返回。
2. Token 失效：复用既有会话失效处理。
3. 文件不存在：提示文件可能已被移动或删除。
4. 格式不支持：提供下载、外部打开、Web 打开。
5. 文件过大：显示限制说明并提供外部打开或下载。
6. 媒体播放失败：提供重试、外部打开和下载。
7. 字幕失败：不阻断主视频播放。
8. 网络错误：提供重试。

### 14.3 崩溃防护

1. 任何预览解析错误不得导致 App 崩溃。
2. Markdown 渲染异常必须降级为源码或错误态。
3. 媒体播放器异常必须释放资源并回到稳定 UI 状态。
4. 外部 Intent 无接收方时必须捕获并提示。

## 15. 安全要求

### 15.1 Token 与 URL 安全

1. Token 不得写入日志。
2. Token 不得拼入可复制、可分享或外部打开的 URL。
3. Authorization Header 不得传给非 OpenList 域名。
4. 签名 URL 不长期缓存。
5. 外部打开不得泄露实例 Token。

### 15.2 Markdown 安全

1. Markdown 渲染不得执行脚本。
2. 不自动执行 HTML 中的危险内容。
3. 远程资源加载失败或被阻止时不影响主内容。
4. 外部链接点击策略需在执行计划中明确。

### 15.3 文件内容安全

1. App 不自动执行下载文件。
2. 外部打开前应由系统选择器或安全 Intent 处理。
3. 临时文件或临时 URI 应具备最小授权范围。
4. 本地缓存应按实例隔离并可被清理。

### 15.4 日志脱敏

1. 预览 URL、媒体 URL、请求头和错误信息必须走既有脱敏策略。
2. 字幕、Markdown 和文本内容不应自动写入日志。
3. 播放失败日志不得包含 Token、Cookie、Authorization 或敏感签名参数。

## 16. Android 权限要求

v0.4 默认不新增危险权限。

要求：

1. 图片、文本、Markdown、视频、音频均优先从网络流读取，不申请全盘存储权限。
2. 外部打开如需临时文件，应使用应用私有缓存和 `FileProvider` 或等价安全机制。
3. 不申请 `MANAGE_EXTERNAL_STORAGE`。
4. 不申请相机、联系人、位置、短信、电话等无关权限。
5. 若执行计划提出新增权限，必须说明原因、影响和验收方式。

## 17. 技术约束

### 17.1 架构约束

1. 继续采用单 Activity + Compose Navigation + MVVM + Repository。
2. UI 层不得直接访问 Retrofit、DAO 或底层 OkHttp。
3. 新增预览模块必须保持依赖方向单向无环。
4. `feature:preview` 不应直接依赖其他 feature 的 UI 实现。
5. 统一文件打开分发应放在可被 files、search、share、task 复用的位置。

### 17.2 依赖约束

1. 媒体播放可引入或正式启用 Media3。
2. 图片加载可引入或正式启用 Coil。
3. Markdown 渲染库需在执行计划中评估安全性、体积和 Compose 适配。
4. PDF/Office 内置渲染相关依赖不得加入 v0.4。
5. 新依赖必须记录用途、影响范围和替代方案。

### 17.3 性能约束

1. 大文本不得一次性无上限读入内存。
2. 图片加载必须避免主线程阻塞。
3. 媒体播放必须释放播放器资源。
4. 列表缩略图不得造成明显滚动卡顿。
5. 预览页退出后应取消未完成请求。

### 17.4 兼容性约束

1. 继续支持项目既定 Android 最低版本。
2. 支持横竖屏切换时不丢失关键播放状态。
3. 子路径部署的实例必须可正常预览。
4. 中文、空格和特殊字符路径必须可正常打开。
5. 不同存储驱动能力差异必须有兜底提示。

## 18. 验收标准

### 18.1 功能验收

1. 文件列表可打开图片预览。
2. 文件详情可打开图片预览。
3. 搜索结果可打开可预览文件。
4. 分享详情可打开已知文件路径。
5. 任务完成项可打开目标文件或跳转目录。
6. 文本文件可预览。
7. Markdown 文件可预览。
8. PDF 可外部打开或提示下载/Web 打开。
9. Office 文件可外部打开或提示下载/Web 打开。
10. 视频可内置播放。
11. 音频可内置播放。
12. 视频字幕入口可用。
13. 不支持格式有明确兜底。
14. 预览失败可重试或返回。
15. Token 失效能回到既有处理链路。

### 18.2 技术验收

1. `feature:preview` 模块边界清晰。
2. UI 层无直接 Retrofit 调用。
3. 所有预览相关 API 经 Repository。
4. 预览缓存带 `instanceId`。
5. 删除实例时清理预览缓存。
6. 路径编码继续复用统一工具。
7. 媒体播放异常不崩溃。
8. Markdown 渲染不执行危险脚本。
9. 外部打开无接收方时不崩溃。
10. 新依赖用途明确且不引入非 v0.4 范围能力。

### 18.3 多实例验收

1. A 实例预览缓存不影响 B 实例。
2. A 实例 Token 不会用于 B 实例预览请求。
3. 删除实例 A 后，其预览缓存被清理。
4. 切换实例后播放器和预览请求不会串实例。
5. 同路径不同实例文件可分别预览。

### 18.4 路径与编码验收

1. 中文文件名可预览。
2. 空格文件名可预览。
3. 特殊字符文件名可预览。
4. 子路径部署实例可预览。
5. 字幕文件路径编码正确。
6. 外部打开路径不因编码错误失败。

### 18.5 UI/UX 验收

2. 不在 PRD 内重复设计令牌，以实现阶段直接引用 `DESIGN.md` 和 `core:designsystem` 为准。
3. 浅色模式可用。
4. 深色模式可用。
5. 横屏视频播放可用。
6. 加载、错误、空态、无权限状态清晰。
7. 触控目标和按钮层级符合现有设计系统。

### 18.6 安全验收

1. 日志不包含 Token。
2. 外部打开不泄露 Authorization Header。
3. 签名 URL 不长期缓存。
4. Markdown 不执行脚本。

## 20. v0.4 成功标准

v0.4 成功的最低标准：

1. 用户可以从文件列表、文件详情、搜索结果、分享详情和任务中心进入统一打开分发。
2. 图片、文本、Markdown、视频、音频具备 App 内打开能力。
3. PDF、Office 和未知格式具备外部打开或下载/Web 打开兜底。
4. 视频和音频播放失败不会导致崩溃。
5. 字幕支持达到基础可用水平。
6. 所有预览和播放能力遵守多实例隔离、Token 安全和路径编码规则。
7. v0.1/v0.2/v0.3 主流程不回归。
8. 新增 UI 遵守 `DESIGN.md`，但不把 `DESIGN.md` 内容复制进需求文档。
9. v0.4 执行计划可直接据此拆分任务、阻塞关系和验收清单。

## 21. 推荐结论

推荐确认 v0.4 范围为：

```text
统一文件打开分发
+ 图片预览
+ 文本预览
+ Markdown 预览
+ PDF/Office 外部打开
+ 视频播放器
+ 音频播放器
+ 基础字幕支持
+ 不支持格式兜底
```

不推荐在 v0.4 同时加入 PDF 内置预览、Office 内置预览、完整分享态浏览、歌词、代码高亮、图片画廊、归档浏览或管理台。那些能力会显著扩大范围，且与 `ROADMAP.md` 的 v0.4 定位不一致。

v0.4 的关键是建立稳定的“文件打开分发”能力：后续 v0.5 管理台和 v1.0 原子级对齐都应复用它，而不是为每个入口重复实现预览逻辑。
