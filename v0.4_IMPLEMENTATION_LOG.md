## S0：验证前置

### DEC-1 结论
`openlist-ref/`（后端参考源码的精简 checkout：`common/`、`handles/`、`middlewares/`、`model/`、`router.go`，31 个 Go 文件，已 gitignore）在仓库中已存在且可用，无需重新克隆。按 DEC-1 走"源码优先"路径，未降级为纯 API 文档兜底。

### V-401 `/api/fs/get` 直链字段
`raw_url` 由 `FsGet` 处理器（`fsread.go`）动态生成：需代理的存储返回 `/p{path}?sign=xxx`（`GenerateDownProxyURL`）；支持直链的存储返回驱动直链（`fs.Link()`/`model.GetUrl()`）。签名仅在存储启用签名/`SignAll` 时按路径生成，重新 `fs/get` 会拿到新签名。签名生成/校验细节（`internal/sign` 包）不在本地 checkout 内，标记 **provisional**。
**客户端结论**：沿用"每次进预览重新 fs/get，不复用旧 URL"的策略（S2 落地）。

### V-402 `/d/*path` 与 `/p/*path`
`/d/*path`：302 重定向到驱动直链（`down.go` `Down()`）。`/p/*path`：服务端反代/流式（`down.go` `Proxy()`）。`raw_url` 优先级复核确认：`fs/get` 内部按 `MustProxy()`/驱动能力决定返回代理路径还是直链，客户端固定"优先用 `raw_url`，为空时才退回 `/d/`+sign"的既有 v0.1 结论成立。`/p/` 代理路径确认是部分驱动（`MustProxy()`/`WebProxy`）的必需路径。

### V-403 Range 支持
`/d/*path` 是 302 重定向，Range 支持取决于目标驱动直链本身。`/p/*path` 反代路径下，`common.ProxyRange()` 按存储配置 `ProxyRange` 布尔值有条件转发 Range 头；该配置为驱动/存储级别，客户端无法预知。**客户端结论**：不假设 Range 一定可用；播放器按 P-412 兜底策略处理，不对 Range 失败做特殊报错，只统一走"播放失败→刷新→外部打开/下载"链路。

### V-404（新增确认，非计划原表格项，但与 V-410 相关）
鉴权失败统一返回 HTTP 状态码（401/403）+ JSON body（`common.ErrorResp`/`ErrorStrResp`），不是重定向或 200+错误体。

### V-405 分享详情已知路径
`sharing.go`：分享详情文件路径导出为分享态 URL（`/sd/{sid}/*path`），**不能**直接用创建者的普通 `fs/get` 访问同一分享态路径字符串；但 `Share.files[]` 在 v0.3 客户端模型中存的是原始文件系统路径（非分享态 URL），创建者本人对该原始路径走普通 `fs/get` 是可行的（权限检查按 `user.BasePath` 前缀匹配）。**客户端结论**：P-406 维持"仅登录用户、以自身权限走普通 fs 链路"不变，分享详情预览入口传入的是原始路径而非分享 URL。

### V-406 字幕候选可靠性
`fs/list`→`ObjResp[]` 含 `name`/`size`/`modified`/`type` 等完整字段，且后端自身已有 `filterRelated()`（`fsread.go`，按去扩展名前缀匹配）证明这套匹配思路成立。**客户端结论**：P-414 的"同目录 fs/list + 文件名前缀匹配"方案确认可靠，源码确认级别。

### V-407 路径编码一致性
逻辑路径（JSON body 中的 `path` 字段）纯文本不编码；下载/代理 URL 的路径段编码由客户端（`OpenListPathCodec`/OkHttp `addPathSegment`）负责，服务端生成 `/p` URL 时用 `utils.EncodePath()` 同规则编码。两端约定一致，v0.1 起的 `OpenListPathCodec` 封装继续适用，无需新增编码分支。

### V-408 / V-409 / V-410
- V-408（Media3 容器支持面）：纯客户端能力问题，不依赖后端源码，S5 实现时按默认 ExoPlayer 解码器能力处理，不支持的格式走 `MediaUnsupported`。
- V-409（Markdown 相对资源解析）：客户端设计问题（`basePath` + `/d/` 构造），S3 实现。
- V-410（签名/Token 失效表现）：`middlewares/auth.go` 确认 Token 失效场景（过期/篡改/密码变更/被禁用/被 invalidate）统一返回 401 JSON；签名验证中间件（`sign.Verify`）具体实现不在本地 checkout 内，**provisional**：客户端按"任何 4xx 播放/预览错误触发一次刷新"处理（P-412），不区分 401/403 细分因果。

### 结论
本地 `openlist-ref` 精简副本覆盖了 fs/get、fs/list、down、sharing、auth 中间件的路由/处理器层逻辑，足以支撑 S0 所需验证结论定稿；`internal/sign`、`server/common`（ProxyRange/Proxy 实现）等内部包不在 checkout 范围内，相关细节标记 provisional，按 DEC-1 兜底口径（防御式解码、安全默认值、不假设成功）处理，不阻塞实现。

### DEC-2/DEC-3
按用户预授权直接采纳：Markdown 渲染库 = Markwon（core 最小引入）；引入 androidx.media3（exoplayer + ui，随实现定版）。版本目录变更将在 S1 落地并在此记录。

### 实现决定
- `PreviewCacheEntity` 用合成 `id` 主键 + `(instanceId,path,kind)` 唯一索引（而非复合主键），便于 REPLACE 式 upsert；`cacheKey` 作为预拼接的查找字符串保留。
- `PreviewSource` 目前只有 `RemoteUrl(url, headersRequired)` 一个变体，按最小化设计，S2/S3 需要本地文件变体时再加。
- Repository 空实现用无参构造函数（S2 起按需加真实依赖），避免过早引入未使用的注入参数。
- Media3 1.4.1 / Markwon 4.6.2 为已知稳定版本，因无法联网核实"当前最新稳定版"，构建阶段如需升级另行处理。
