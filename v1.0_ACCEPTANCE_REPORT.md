## 8. S7 Stabilization: Two Independent Audits

与 v0.5 报告"两个独立审计均零发现"不同，**本轮两个审计各发现 1 项真实问题，均已修复并补测**——审计的价值在于发现问题而非零发现本身，这被视为审计有效性的正面信号。

### 8.1 审计一：安全脱敏面

**结论：发现 1 项确认漏洞，已修复**

- `SafeLogger.redact()` 原实现要求敏感 key 紧邻 `:`/`=` 分隔符，JSON 引号键形式（`"password":"value"`）因 key 与分隔符之间隔着引号字符而完全不匹配。`OpenListClientFactory` 在用户开启"调试日志"设置时以 `HttpLoggingInterceptor.Level.BODY` 记录完整请求/响应体——意味着 v1.0 新增的登录/OTP/分享密码请求体在调试日志打开时会明文完整落 logcat。同一审计还发现 `otp_code`/`otpCode` 字段完全不在脱敏关键字清单内，无论 JSON 修复与否 OTP 值都不会被脱敏。
- **修复**：key 匹配两侧允许标识符前后缀（`otp_code`/`access_token` 等真实字段名按子串命中）；值捕获改为完整 JSON 引号字符串或最多两词，同时修正了"`Authorization: Bearer <token>` 只脱敏 `Bearer` 一词、真正 token 未被遮蔽"的历史脆弱点；新增 `otp` 关键字。`SafeLoggerTest` 由 5 项增至 7 项，全部通过。
- 审计确认无问题的面：OTP/分享密码全程只存在于 ViewModel 内存态；Markdown 图片外链结构性不经过 `AuthInterceptor`；`raw_url`/`sign=` 值在本轮新增 6 个 Repository 中均无直接日志调用；管理任务批量确认对话框不携带敏感负载。

### 8.2 审计二：缓存与多实例隔离 + 权限门控回归面

**结论：发现 1 项 UX 一致性问题（非安全漏洞），已修复**

- `FileListViewModel` 的批量选择工具栏只在进入多选模式时校验 `canWrite`，多选期间若目录能力因新鲜数据到达从"乐观 true/unknown"降级为"确认 false"，批量操作按钮不会跟着失效。后端 403 仍是最终裁决，故非安全绕过，但存在"UI 允许点击、后端拒绝"的体验落差。
- **修复**：`navigateTo` 的 `Fresh` 分支新增判定，能力降级为确认 false 且当前处于多选模式时自动退出多选。因 `:feature:files` 模块此前无单元测试基础设施（仅声明 `junit`），本次以代码审查+真机冒烟佐证，未新建测试工程（记入日志而非静默变更）。
- 审计确认无问题的面：分享入站解析结果零 DAO 写入；`DirectoryCapability` 未入 Room 且跨目录正确重置；`deleteInstance` 级联覆盖本轮新增的 `remoteTaskDao`/`adminCacheDao`；管理台 Tab 结构性挂载门控覆盖本轮新增入口。次要观察（已记入代码注释）：Markdown 内嵌图片走 Markwon 自身 `NetworkSchemeHandler`，完全不经过 Coil，因此不存在假设中的"跨实例 Coil 缓存 key 冲突"风险，但也意味着零磁盘/内存缓存。
