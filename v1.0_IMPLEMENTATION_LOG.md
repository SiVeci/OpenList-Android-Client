**V-604（`fs/list` 目录级 write 语义）—— 复核完成，语义比初步结论更细，直接影响 S1 门控设计**
- `FsListResp.write`（`server/handles/fsread.go:122`）**只由 `common.CanWrite(user, meta, path)` 计算**（meta 白名单机制：`meta.WriteUsers`/`WriteUsersSub`，`server/common/check.go:32-41`），**不直接读用户 `Permission` 位**。
- 但实际"是否允许上传"的判定更严格：`fsread.go:96` 的 `canWriteContentAtPath := common.CanWrite(user, meta, reqPath) && (user.CanWriteContent() || common.CanWriteContentBypassUserPerms(meta, reqPath))`——**同时需要 meta 白名单 write 通过 `且` 用户位 `CanWriteContent()`（或 meta 级 bypass）**。
- **重要偏差**：响应体 `write` 字段本身**只反映 meta 白名单结果，不反映用户 `CanWriteContent()` 位**。这意味着客户端不能只用 `write` 单一字段做门控——`write=true` 但用户无 `CanWriteContent()` 位时，上传仍会被后端拒绝。**S1 设计调整**：`DirectoryCapability.canWrite` 必须组合 `write ∧ session.canDo(PERM_WRITE_CONTENT)`（PRD §4.2.B.5 本就要求"目录级 write + 会话权限位"组合，源码复核证实这个组合是必需的，不是保守冗余）。
- 无其他细分能力字段（重命名/删除等无目录级标志），`write` 是唯一的目录能力字段。
- 分享态（`SharingList`，`server/handles/sharing.go:102`）**硬编码 `Write:false`**，与分享态只读的假设一致。

## 需要你决策：DEC-607 raw_url 端口加固是否仍纳入 v1.0

### 背景

v1.0 执行计划 §16 DEC-607 基于此前观察到的"nas.siveci.com:18443 反代环境下 fs/get 返回的 raw_url 丢失端口，导致预览/下载误连到 Unraid 的 :443 登录页"设定，推荐方案 A：App 侧加固（host 相同、端口不同时重写为实例端口 + 预览响应 content-type 粗校验）。

### 为什么现在必须决策

本轮 S0 用该实例做匿名 `fs/list`/`fs/get` 实测（root 目录、`local` 本地存储、`workspace` WebDav 存储、`pikpak` 云存储四个不同 provider 挂载点），**全部返回的 `raw_url` 均正确带 `:18443` 端口**，且实际 `curl` 该 URL 能拿到正确文件内容。这与历史记忆中的"丢端口"现象不符，判断是实例的服务端 `site_url` 配置已在两次会话之间被修正。这影响 S1 是否需要实现"host 匹配、端口重写"这部分代码——如果症状已不存在，这部分加固代码将没有可复现的场景来验证，且是"为一个当前不存在的问题写防御代码"。

注：DEC-607 的另一半（预览/播放响应 content-type 粗校验，用于识别裸错误页）**不受此影响**——S0 已独立复现"直链类端点在越界 Range/其他异常请求下返回裸文本而非 JSON"的问题（416/405），且 S1-T1 的 `ErrorMapping.HttpException` 修复已经覆盖这类响应在 Retrofit 层的正确映射。

### 选项

A. **仍按计划纳入完整 DEC-607 加固**（端口重写 + content-type 粗校验）
* 优点：面向"其他用户的反代实例仍可能有同样的 site_url 配置错误"这一类问题做通用防御；实现成本小（一处 URL 后处理）；即使当前实例不再复现，未来该实例或其他实例的配置回退/新实例部署仍可能触发。
* 缺点：端口重写部分当前**没有可复现场景**做真实回归验证，只能靠单测模拟"host 相同端口不同"的构造用例，不能在真实环境里跑通"确实避免了错误行为"的闭环证据；有把"防御未知配置错误"当成既定验收项的风险（v1.0 PRD 强调真实回归证据优先）。
* 影响：S1-T1 范围新增一处 URL 处理逻辑 + 对应单测，不影响其他模块。

B. **只做 content-type 粗校验（识别裸错误页），不做端口重写**
* 优点：只保留有真实场景复现依据的那部分（416/405 裸文本响应），其余交给 `HttpException` 分支统一处理；避免为不再复现的问题写代码。
* 缺点：如果确实还有其他用户/未来这个实例的 site_url 重新配错，App 不会自动纠正端口，用户会看到预览/下载失败（但至少会有可读错误提示，而非静默连错端口）。
* 影响：S1-T1 范围缩小，`KNOWN_ISSUES.md`/`Parity_Matrix.md` 记录"raw_url 端口问题为环境配置类，此前观察到过，当前实例已修复，App 不做静默纠正，只提供可读错误"。

### 推荐方案

我推荐选择 **B**：真实回归证据是本轮反复强调的口径（"不得以静态审查代替实测"），既然当前找不到可复现场景来验证端口重写代码的正确性，为一个不可复现的问题写生产代码不符合这个原则；content-type 粗校验部分已有真实证据支持，继续做。如果用户/后续会话在其他实例上重新复现丢端口问题，可以在那时用真实证据重新开这一项。

### 用户决策

**采纳方案 B**：只做 content-type 粗校验（识别裸错误页），不做 raw_url 端口重写。理由与推荐一致——无可复现场景验证端口重写代码，真实回归证据优先。DEC-607 最终形态：S1-T1 的 `HttpException` 修复覆盖裸错误响应识别；`KNOWN_ISSUES.md`/`Parity_Matrix.md` 记录 raw_url 曾观察到丢端口问题、当前实例已自行修复、App 不做静默端口纠正。

### S7-T2 审计一：安全脱敏面 —— 发现并修复 1 项确认漏洞

**确认漏洞（已修复）**：`SafeLogger.redact()`（`core/common/.../SafeLogger.kt`）的正则要求敏感 key 文本紧邻分隔符 `:`/`=`，JSON 引号键形式 `"password":"value"` 因 key 与分隔符之间隔着一个 `"` 而完全不匹配——`OpenListClientFactory` 在用户开启"调试日志"设置时会以 `HttpLoggingInterceptor.Level.BODY` 记录完整请求/响应体，意味着本轮新增的登录/OTP/分享密码请求体在调试日志打开时会**明文完整落 logcat**。同一次审计还发现 `otp_code`/`otpCode` 字段根本不在 `sensitiveKeyPatterns` 清单里，OTP 值完全未脱敏（无论 JSON 修复与否）。

**修复**：`redact()` 的每个 key 正则改为 `"?[A-Za-z0-9_]*(?:KEY)[A-Za-z0-9_]*"?\s*[:=]\s*(quoted-json-string|token+可选第二词)`——(1) key 两侧允许可选引号与标识符前后缀（同时让 `otp_code`/`access_token` 这类真实字段名按子串命中，不要求与 key 完全相等）；(2) 值改为"整段 JSON 引号字符串（处理转义）或最多两个空格分隔词"，修复了原实现"`Authorization: Bearer abc123` 只脱敏 `Bearer` 一词，真正的 token `abc123` 未被遮蔽"的脆弱点（旧版 `SafeLoggerTest` 曾把这一行为当"预期"写死断言，本轮已更正该测试）；(3) 新增 `otp` 关键字覆盖 OTP 字段。`\bsign\b` 的单词边界语义不受影响（验证：`\b` 在连续字母串内部永远不匹配，`design`/`assignment` 测试仍通过）。

**单测**：`SafeLoggerTest` 由 5 项增至 7 项：重写"Bearer 两词值整体脱敏"断言（原先文档"已知缺陷"的用例改为断言正确行为）；新增 JSON 引号键请求体脱敏（`password`/`otp_code` 均命中，非敏感字段 `username` 保留）；新增嵌套 JSON `token` 字段脱敏。全部通过。

**审计确认无问题的面**（逐项复核，非默认通过）：OTP/分享密码全程只存在于 ViewModel `MutableStateFlow`（不入 Room、`SavedStateHandle` 只带非敏感 `instanceId`）；Markdown 图片外链请求结构性不经过 `AuthInterceptor`（`NetworkSchemeHandler`/`DataUriSchemeHandler` 与既有认证 Retrofit 客户端是两条独立路径）；`raw_url`/`sign=` 值在本轮新增的 6 个 Repository 中均无直接 `Log`/`Timber` 调用；管理任务批量确认对话框不携带敏感负载。
