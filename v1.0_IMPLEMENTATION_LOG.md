**V-604（`fs/list` 目录级 write 语义）—— 复核完成，语义比初步结论更细，直接影响 S1 门控设计**
- `FsListResp.write`（`server/handles/fsread.go:122`）**只由 `common.CanWrite(user, meta, path)` 计算**（meta 白名单机制：`meta.WriteUsers`/`WriteUsersSub`，`server/common/check.go:32-41`），**不直接读用户 `Permission` 位**。
- 但实际"是否允许上传"的判定更严格：`fsread.go:96` 的 `canWriteContentAtPath := common.CanWrite(user, meta, reqPath) && (user.CanWriteContent() || common.CanWriteContentBypassUserPerms(meta, reqPath))`——**同时需要 meta 白名单 write 通过 `且` 用户位 `CanWriteContent()`（或 meta 级 bypass）**。
- **重要偏差**：响应体 `write` 字段本身**只反映 meta 白名单结果，不反映用户 `CanWriteContent()` 位**。这意味着客户端不能只用 `write` 单一字段做门控——`write=true` 但用户无 `CanWriteContent()` 位时，上传仍会被后端拒绝。**S1 设计调整**：`DirectoryCapability.canWrite` 必须组合 `write ∧ session.canDo(PERM_WRITE_CONTENT)`（PRD §4.2.B.5 本就要求"目录级 write + 会话权限位"组合，源码复核证实这个组合是必需的，不是保守冗余）。
- 无其他细分能力字段（重命名/删除等无目录级标志），`write` 是唯一的目录能力字段。
- 分享态（`SharingList`，`server/handles/sharing.go:102`）**硬编码 `Write:false`**，与分享态只读的假设一致。

## 需要你决策：DEC-607 raw_url 端口加固是否仍纳入 v1.0

### 背景

v1.0 执行计划 §16 DEC-607 基于此前观察到的"nas.siveci.com:18443 反代环境下 fs/get 返回的 raw_url 丢失端口，导致预览/下载误连到 Unraid 的 :443 登录页"设定，推荐方案 A：App 侧加固（host 相同、端口不同时重写为实例端口 + 预览响应 content-type 粗校验）。

### 为什么现在必须决策

本轮 S0 用该实例做匿名 `fs/list`/`fs/get` 实测（root 目录、`local` 本地存储、`workspace` WebDav 存储、`pikpak` 云存储四个不同 provider 挂载点），**全部返回的 `raw_url` 均正确带 `:18443` 端口**，且实际 `curl` 该 URL 能拿到正确文件内容。这与历史记忆中的"丢端口"现象不符，判断是实例的服务端 `site_url` 配置已在两次会话之间被修正。这影响 S1 是否需要实现"host 匹配、端口重写"这部分代码——如果症状已不存在，这部分加固代码将没有可复现的场景来验证，且是"为一个当前不存在的问题写防御代码"。

注：DEC-607 的另一半（预览/播放响应 content-type 粗校验，用于识别裸错误页）**不受此影响**——S0 已独立复现"直链类端点在越界 Range/其他异常请求下返回裸文本而非 JSON"的问题（416/405），且 S1-T1 的 `ErrorMapping.HttpException` 修复已经覆盖这类响应在 Retrofit 层的正确映射。

### 选项

A. **仍按计划纳入完整 DEC-607 加固**（端口重写 + content-type 粗校验）
* 优点：面向"其他用户的反代实例仍可能有同样的 site_url 配置错误"这一类问题做通用防御；实现成本小（一处 URL 后处理）；即使当前实例不再复现，未来该实例或其他实例的配置回退/新实例部署仍可能触发。
* 缺点：端口重写部分当前**没有可复现场景**做真实回归验证，只能靠单测模拟"host 相同端口不同"的构造用例，不能在真实环境里跑通"确实避免了错误行为"的闭环证据；有把"防御未知配置错误"当成既定验收项的风险（v1.0 PRD 强调真实回归证据优先）。
* 影响：S1-T1 范围新增一处 URL 处理逻辑 + 对应单测，不影响其他模块。

B. **只做 content-type 粗校验（识别裸错误页），不做端口重写**
* 优点：只保留有真实场景复现依据的那部分（416/405 裸文本响应），其余交给 `HttpException` 分支统一处理；避免为不再复现的问题写代码。
* 缺点：如果确实还有其他用户/未来这个实例的 site_url 重新配错，App 不会自动纠正端口，用户会看到预览/下载失败（但至少会有可读错误提示，而非静默连错端口）。
* 影响：S1-T1 范围缩小，`KNOWN_ISSUES.md`/`Parity_Matrix.md` 记录"raw_url 端口问题为环境配置类，此前观察到过，当前实例已修复，App 不做静默纠正，只提供可读错误"。

### 推荐方案

我推荐选择 **B**：真实回归证据是本轮反复强调的口径（"不得以静态审查代替实测"），既然当前找不到可复现场景来验证端口重写代码的正确性，为一个不可复现的问题写生产代码不符合这个原则；content-type 粗校验部分已有真实证据支持，继续做。如果用户/后续会话在其他实例上重新复现丢端口问题，可以在那时用真实证据重新开这一项。

### 用户决策

**采纳方案 B**：只做 content-type 粗校验（识别裸错误页），不做 raw_url 端口重写。理由与推荐一致——无可复现场景验证端口重写代码，真实回归证据优先。DEC-607 最终形态：S1-T1 的 `HttpException` 修复覆盖裸错误响应识别；`KNOWN_ISSUES.md`/`Parity_Matrix.md` 记录 raw_url 曾观察到丢端口问题、当前实例已自行修复、App 不做静默端口纠正。

## S1 登录与权限补齐

### 实现内容

**S1-T1 `ErrorMapping` HttpException 修复**（`core/network/ErrorMapping.kt`）：`Throwable.toDomainError()` 新增 `is HttpException` 分支，按 `code()` 复用既有 `codeToDomainError` 语义，message 取 `errorBody()`（读取失败/为空时回退 `"HTTP {code}"`）。不再落入 `Unknown`。单测 `ErrorMappingTest`（7 项：401/403/404/5xx/416 文本体/405 空体/确认不再是 Unknown）全绿。

**S1-T2 登录模型**：`Session.AuthType` 新增 `LDAP`（String 列存储，零迁移）；新增 `core:model/LoginResult.kt`（`Success(session)`/`NeedOtp(challenge)`，**不含 Failure 分支**——刻意偏离计划 §8 原文里"Failure 包 DomainError"的表述，因为 `LoginResult` 若要携带 `DomainError` 需要 `core:model` 新增对 `core:common` 的模块依赖；改为在 Repository 边界用既有 `ApiResult.Failure` 承载失败，`LoginResult` 只表达"成功态的两种形态"，不引入新模块依赖，效果等价，已按 AGENTS.md 精神记录此处偏差）；新增 `OtpChallenge`（仅内存态：instanceId/method/username，不含密码）。`DomainError` 新增 `AuthMethodUnavailable`/`OtpInvalid`（`OtpRequired` 未新增——NEED_OTP 用 `LoginResult.NeedOtp` 表达，不是错误态，同样是对计划 §12.1 的一处务实偏差）。

**S1-T3 `AuthRepositoryImpl`**：`loginWithPassword` 增加 `otpCode: String? = null` 参数，返回类型升级为 `ApiResult<LoginResult>`；首次 402（`otpCode` 为空）→ `LoginResult.NeedOtp`，重试仍 402（`otpCode` 非空）→ `DomainError.OtpInvalid`（避免重复弹出"需要 OTP"造成用户困惑）。新增 `loginWithLdap`（`OpenListApi.loginLdap` → `POST api/auth/login/ldap`，复用 `LoginReq`）；V-601 确认 LDAP 无 OTP 分支，故不接受 `otpCode` 参数。LDAP 403（"ldap is not enabled"/"login via ldap is not allowed"）在 Repository 层单独识别为 `AuthMethodUnavailable`（不复用共享 `codeToDomainError`，因其会把这两种 403 都吞成通用 `Forbidden` 并丢失 message——沿用 v0.5 `safeAdminOperationCall` 的"共享映射不够用时本地覆盖"先例）。单测 `AuthRepositoryImplTest`（10 项：密码成功/首次 402→NeedOtp/重试 402→OtpInvalid/401、LDAP 成功/两种 403→AuthMethodUnavailable/400 不越权映射、guest/token 回归）全绿。

**S1-T4 登录页 UI**：`LoginViewModel`/`LoginScreen` 重构为 `LoginMethod{PASSWORD,LDAP,TOKEN}` 三态 Tab（复用 `AdminTabRow`，不新建组件）+ 游客独立按钮（四方式区分：账号密码/LDAP/Token/游客）；OTP 走内联第二步（DEC-601，同 ViewModel 状态机，`otpChallenge` 非空即切到验证码输入视图，`cancelOtp()` 可退回）；页面底部加 SSO/WebAuthn 纯文案兜底提示（不做流程，指引改用 Token 登录，PRD §4.2.B.4）。OTP 码只存在 `_uiState`（进程内存），不落盘不进日志。

**S1-T5 `DirectoryCapability` + 门控**：新增 `core:model/DirectoryCapability.kt`（`canWrite: Boolean?`，null=unknown）。`FileListResult.Cached`/`Fresh` 新增 `capability` 字段（Cached 默认 UNKNOWN，Fresh 由仓储计算）。`FilesRepositoryImpl` 注入 `AuthRepository`，新增 `resolveCapability()`：**`canWrite = fsListResp.write && session.canDo(PERM_WRITE)`**——这是 V-604 的核心发现：`write` 字段本身只反映服务端 meta 白名单结果，不反映用户 `CanWriteContent` 权限位，必须两者都为真。单测 `FilesRepositoryImplTest`（5 项：write=true+有位→true；write=true+无位→false【V-604 关键回归】；write=false→false；无 session→false；缓存态→unknown）全绿。

**S1-T6 文件列表门控接线**：`FileListUiState` 用 `isGuest: Boolean` + `directoryCapability: DirectoryCapability` 替换原先纯 session 驱动的 `canWrite` 字段；新增计算属性 `canWrite`（`!isGuest && capability.canWrite != false`，true/unknown 都乐观展示、确认 false 才隐藏）与 `writeCapabilityUnknown`（供未来低风险文案使用）。跨目录导航时重置 `directoryCapability` 为 UNKNOWN，避免旧目录的权限状态閃现到新目录。游客隐藏逻辑不变（仍是硬隐藏，不因 capability 变化）。

### 偏离计划的记录

1. `LoginResult` 未按计划放 `core:model` 且不含 `Failure` 分支（见上 S1-T2），是模块依赖最小化的务实选择，效果等价。
